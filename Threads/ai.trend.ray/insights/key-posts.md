## AI 운영체제를 만들려는 글

### 1. 문과다
```text
# 문과다

문과다. 
제약 영업만 10년했다. 
바이브코딩으로 서비스 만들고 있다. 
좌충우돌시간 1년째. 
AI로 인생 후반 준비하려고 혼자 발버둥치고 있다. 

개발자 언어는 모른다. 
근데 좌충우돌으로 얻은 경험을 풀테니 개발1도 모르는 귀있는 사람은 그냥 들어라.

1. 오픈클로 - 다 해줄거라고 생각하지말자. 딸깍으로 절대 끝나지 않는다. 

옵시디언으로 문서정리하고 문서 바탕으로 애이전트와 소통하려고 했으나 문서 정리하다 끝날 가능성 95%. 

서비스 만드는걸 오픈클로 선택했다면 오픈클로 환상을 깨라.
```
왜 중요한가: 계정의 출발점이 가장 솔직하게 드러나는 자기소개 글이다. 비개발자, 영업 출신, AI로 인생 후반을 준비하는 사람이라는 서사가 이후 모든 실험의 배경이 된다.

### 2. 오픈클로를 다시 깔고
```text
# 오픈클로를 다시 깔고

오픈클로를 다시 깔고
환경 세팅부터 다시 시작.
openclaw.json 파일부터 뜯어 보고
agents.md, soul.md, memory.md 세 가지
뜯어보면서 다시 설정.

옵시디언으로 
운영 OS로 쓸 때 세팅 사항.

1. 문서 frontmatter 표준화
2. 기준본(source of truth) 단일화
3. Home 문서 하나로 진입점 통일
4. Inbox → Brief → Run → Decision 루프 고정
5. Daily 로그는 짧게라도 반드시 남기기
6. 장기 기억(Memory)과 일일 기록(Daily) 분리
7. 프로젝트 인덱스만 만들고, 실물 프로젝트는 따로 둔다
8. Archive 규칙을 미리 정하기
9. 에이전트 역할표를 문서화
10. “읽기 규칙”과 “쓰기 규칙”을 따로 둔다

그리고, 오픈클로한테 물어볼때는

1. 
하네스 구조가 좋다고하여
'하네스 구조어때? 좋은것 같아?'
라고 물어보면

'하네스 구조 좋다. 이쪽으로 진행 계획을 짜보겠다'
라는 동조의 결과물이 나올 가능성이 높다.

2.
그래서 
이렇게 질문하면
하네스 구조가 지금 필요한지
안필요한지 오픈클로가 판단한다.

'하네스 구조를 채택하면 좋을지
너가 판단해서 비교해줘.

3.
나도 잘 모르기 때문에
그냥 오픈클로한테
'생산적인 방향으로 선택해'
'너는 어떻게 평가해?'

라고 물어보고 
오픈클로의 말을 듣는다.
```
왜 중요한가: 이 계정의 운영 철학이 가장 집약된 글이다. 문서 표준화, source of truth, 로그, 역할 분리 같은 구조적 집착이 한눈에 보인다.

### 3. 헤르메스 에이전트로
```text
# 헤르메스 에이전트로

헤르메스 에이전트로
회사를 설립하고 
부서를 신설하여 조직을 세우고
각각의 일을 시켰다.

그 장점은 아래와 같다.

1.
다양한 업무를 한꺼번에 던져줘도
AI가 각각의 업무를
각각의 부서에 잘 정리해서 진행시킨다.

오픈클로와는 다르게
헤르메스는 각 부서를 잘 운영하는 듯한 느낌을 받는다

2. 
부서는 서로 연결되어 있다.
서로 유기적으로 움직이기 위해 촘촘하게 짜여있다.
(내 생각과 현재 총괄인 코코AI의 생각은 그렇다)
일단 기본적인 환경셋팅을
다른 프로젝트가 들어오더라도
흐름이 깨지지 않고 흡수할 수 있다.

3. 
지식 에셋 부서는
생각보다 중요했다.
내가 따라가지 못하는 AI 기술 업데이트를
지식 에셋 부서가 
스스로 돌며 찾고
회사에 파일럿으로 돌릴지 스스로 점검한다.
또한 지식 에셋을 쌓아두면
미래 적용점이 많아 진다는 사실.
```
왜 중요한가: "AI를 회사처럼 운영한다"는 세계관이 가장 직접적으로 드러난다. 부서, 총괄, 지식 에셋 팀 같은 표현이 이후 글의 기본 프레임이 된다.

### 4. 헤르메스 + 옵시디언을
```text
# 헤르메스 + 옵시디언을

헤르메스 + 옵시디언을 
OS로 운영할 때

반드시
동기화 에이전트 를 운영하세요.

그래야 문서들이 깨지지 않고
지속적으로 OS가 잘 굴러갑니다.

아래는 코덱스가 전체 검수하면서
칭찬해준 점

* 코덱스 코멘트
"잘된 점: “운영동기화팀”을 별도 역할로 둔 건 아주 좋습니다. 
문서 OS가 커질수록 이 팀이 실제 유지보수 엔진이 됩니다."
```
왜 중요한가: 단순 자동화가 아니라 "유지보수 가능한 운영체제"를 만들려는 시선이 보인다. 동기화와 유지보수를 독립 기능으로 본다는 점이 핵심이다.

### 5. 코덱스 + 옵시디언 + OMX(on my codex)
```text
# 코덱스 + 옵시디언 + OMX(on my codex)

코덱스 + 옵시디언 + OMX(on my codex)
초기 세팅 및 설치 가능하며 클로드가 아래와 같은 장점을 써주었습니다.

//** 코코는 제 헤르메스 애칭입니다 **//

Codex CLI + OMX 설치 시 좋은 점
현재 vs 설치 후 비교
현재: 코코가 모든 걸 하나의 대화창에서 처리
  코딩도 조사도 검수도 같은 컨텍스트에서 순차적으로
설치 후: 코코가 코딩 작업을 Codex에 "던지고" 다른 일 가능
  Codex가 독립 프로세스로 코드를 작성·수정·커밋

실용적 장점 3가지 (아래에서 계속)

① 코딩 작업을 완전히 분리할 수 있다

bash
# 코코가 터미널 한 줄로 코딩 위임
codex exec --full-auto "쿠팡 크롤러에 재시도 로직 추가해줘"
# 코코는 그 동안 대표님과 다른 대화 가능
# Codex는 별도 프로세스에서 코드 수정 → 커밋까지 자동

② 병렬 코딩이 가능하다

bash
# 이슈 3개를 동시에 수정
codex exec "Fix issue #1" & ← 각각 독립 프로세스
codex exec "Fix issue #2" &
codex exec "Fix issue #3" &
# 3개가 동시에 돌아감
현재 Hermes delegate_task는 최대 3개 동시 실행 가능하지만 같은 모델·같은 세션 안이라 무겁습니다. Codex는 완전히 독립된 OS 프로세스라 훨씬 가볍습니다.

③ OMX가 워크플로우를 자동 추적한다

.omx/state/ ← 현재 어떤 작업이 진행 중인지
.omx/logs/ ← 어떤 파일을 수정했는지
.omx/hooks/ ← 작업 완료 시 자동 알림
코코가 "지금 Codex 뭐 하고 있어?"를 물으면 상태를 바로 확인 가능합니다.
```
왜 중요한가: 도구 소개를 넘어, AI 노동을 프로세스 분리와 병렬 처리의 관점으로 보는 글이다. 이 계정이 왜 에이전트를 "조직"처럼 다루는지 잘 보여준다.

## 인간 언어를 AI 언어로 바꾸는 글

### 6. AI한테 일을 잘 시키는 사람들 보면
```text
# AI한테 일을 잘 시키는 사람들 보면

AI한테 일을 잘 시키는 사람들 보면
질문을 잘하는 게 아니라

AI가 대답하기 쉬운 단위로 잘 자릅니다.

“이거 해줘”보다
“1안/2안으로 나눠. 먼저 판단 기준부터 줘.”
이렇게 자르면 퀄이 확 달라집니다.
```
왜 중요한가: 이 계정의 프롬프트 철학을 가장 짧고 정확하게 요약한 글이다. 화려한 문장보다 작업 분해가 중요하다는 믿음이 압축되어 있다.

### 7. AI와 인간은
```text
# AI와 인간은

AI와 인간은
사용 언어가 다르고 정의하는게 맞다.

즉,
한국어(또는 영어)라는 '기호(어휘)'는 공유하지만, 
소통을 관장하는 '사고의 패러다임(구조)'이 완전히 다르다

일반적인 직장인들은 이렇게 말한다.

"마케팅팀에 콘텐츠 기획 담당자를 두고, 
브랜드 가이드에 맞춰 초안을 작성하게 한다. 
다만 대외 발행 전에는 팀장 검토가 필요하다"

하지만

AI에게는 이렇게 말해야 정확하게 이해를 한다

"콘텐츠 기획 역할의 에이전트를 생성하고, 브랜드 가이드와 채널별 작성 규칙을 컨텍스트로 연결한다. 출력은 초안 상태로 제한하며, 게시 API 실행 전 인간 승인 게이트를 둔다."
```
왜 중요한가: 인간 언어와 AI 언어의 차이를 구조 수준에서 설명하는 대표 글이다. 계정 전체를 관통하는 "회사의 언어를 실행 구조로 번역하라"는 문제의식이 선명하다.

### 8. 예를 들어 누군가 회사에서 이렇게 말한다고 하자
```text
# 예를 들어 누군가 회사에서 이렇게 말한다고 하자

예를 들어 누군가 회사에서 이렇게 말한다고 하자.

“이 건 좀 잘 처리해 주세요.” 
사람은 대충 알아듣는다. 
왜냐하면 그 문장에는 실제로는 말해지지 않은 수많은 변수들이 붙어 있기 때문이다.

즉 사람은 문장을 해석하는 것이 아니라, **문장을 둘러싼 맥락을 복원**한다.

그래서 dry한 언어만 놓고 보면 인간 소통은 사실 매우 헐겁다. 
평소에 그것이 잘 굴러가는 이유는 인간이 언어 밖의 층위를 동시에 읽기 때문이다.

예를 들어 대표가 묻는다.

“이거 지금 발행해도 돼?” 
어떤 팀장이 “네”라고 답했다면, 그 “네”는 사실 아래와 같은 암묵적 if-else를 포함한다.

- 브랜드 가이드 위반이 없고
 
- 법무 이슈가 없고
 
- 예산 한도 안이고
 
- 현재 승인 라인을 통과했고
 
- 외부 일정상 지금 발행하는 게 맞다면
 
- 발행 가능
```
왜 중요한가: 단순한 감상이 아니라 꽤 정교한 조직론, 의사소통론으로 확장되는 지점이다. 인간 조직의 암묵지를 AI 운영 설계로 옮기려는 사고가 보인다.

### 9. 비개발자 출신, 문과 출신들이
```text
# 비개발자 출신, 문과 출신들이

비개발자 출신, 문과 출신들이
개발자의 언어를 배우고
개발자의 개념을 탑제한 후
바이브코딩을 하면
금상첨화다. 
이보다 좋은게 어디있겠는가

하지만 이건 불타는 열정을 가진
소수의 사람들이 할 수 있는 일이다
나같은 범부에게는
아주 개발에 대한 아주 미미한 이해와
시도들만 하고 있을 뿐이다.

비개발자가 AI 에이전트를 운영하면
반드시 언어의 간극을 체감하게 된다.
그 간극을 극복하기 위해선
2가지 중 하나다

1., 
개발자의 언어를 배운다

그래야 정확한 '지시'가 가능하다.
개발자의 로직과 개념을 탑제해야
그 이후로 원하는 결과물을 쌓아갈 수 있다.

여기서부터 고통이 시작된다.
나의 언어로 지시하면,
엉뚱한 결과물이 나오던가
혹은 부족한 결과물이 나오기 일쑤다

2.
나의 언어를 가르친다

내가 말하는 스타일을 익히게 하는 것이다.
헤르메스 에이전트가 급상승한 이유는
분명 비개발자들의 언어를 기억하고
학습하는 기능이 있기 때문일 것이다.

하지만 이것로도 부족하다.
어디서부터 어떻게 가르쳐야 할지 알지 못한다.
이를 위해 지식체계를 구축하고
문서화하지만,

매번 충돌이 일어나고,
문서가 비대해지면 어느 지점부터는
맥락과 갈피를 반드시 잃어버리게 된다.

나도 헤르메스 + 옵시디언위키를 통해
에이전트 회사 운영 OS를 만들어 보는 
시도를 계속 하고 있으나

어느 시점에 가면 더 이상 진전하기가 어렵다.

3.
왜 에이전트 회사를 먼저 세팅하는
시도를 하고 있냐면

다양한 프로젝트를 운영할 수 있는 기반을
만들기 위해서이다.

프로젝트별로 진행하면
분명 빠르게 실험결과물(MVP)가 나오겠지만
지속가능하게 굴러가진 않는다.

그렇기 때문에 반드시 '회사처럼'
프로젝트를 만들고
프로젝트를 운영할 수 있는 체제를
만들고 싶은 것이다.
어쩌면 '자비스'를 원하는 것일 수도 있다.
```
왜 중요한가: 비개발자 사용자의 한계와 욕망이 가장 입체적으로 드러난 장문이다. 이 계정이 단순 툴 후기가 아니라 언어 전환 프로젝트라는 점을 보여준다.

## 아이디어, 콘텐츠, 시장을 보는 글

### 10. 또 쓰레드에서 영감을 받습니다
```text
# 또 쓰레드에서 영감을 받습니다

또 쓰레드에서 영감을 받습니다.
어느 분이 

바이브코딩을 잘하려면
도메인의 지식을 언어화하는 능력이 핵심
이라고 하셨는데,
미묘하게 다른 표현에
아이디어가 문뜩 떠오릅니다.

그래서

1. 
제가 아는 전문 지식을 ChatGPT에게 줘봤습니다.
제약영업 사원의 워크플로우에 대해서 이야기 해보랬더니
꽤나 흐름을 잘 이야기 합니다.

하지만 핵심은 여기에 있습니다.
여기서 무엇이 도메인 지식을 필요로 하는지
```
