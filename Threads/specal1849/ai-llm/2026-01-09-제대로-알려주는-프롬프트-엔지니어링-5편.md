# [제대로 알려주는 프롬프트 엔지니어링(5편)]

[제대로 알려주는 프롬프트 엔지니어링(5편)]
롱테일 문제와 평균회귀문제

제가 강의에서 자주하는 말이있습니다.

AI는 전지전능한게 아니라 
그냥 데이터 기반 계산기로 
수많은 데이터를 통해 반영하는 예측/생성 시스템이다.

Chat gpt 나 제미나이 같은 llm은 대부분은 잘 맞는데
최신 정보를 반영 못하거나 
평균적인 대답을 내놓는 경우가 많은데요

이걸 이제 알맹이는 없다 표현하기도 하는데
이유는 대체로 한 가지로

예외 상황이 학습 데이터에서 저빈도로 취급되기 때문입니다.

말이어렵죠? 오늘은 이 이야기를 해보죠.

AI를 더 믿게 만드는 게 아니라
어디서 덜 믿어야 하는지를 분포 관점에서 분석해 봅시다.

1. 실패는 어려운 문제보다 예외 조건에서 발생한다

실무에서 AI를 쓸떄 크게 틀리는 경우는
난이도가 높은 문제라기보다 예외 조건이 많습니다. 

어려운 질문은 사람마다 정의가 다르겠지만 
일반적으로 많이 틀리는 조건을 정리해보면

1.고객 CS의 희귀 컴플레인
2.계약/정책의 예외 조항
3.장애 대응의 드문 에러코드
4.창작 세계관의 미세한 설정 충돌 
5.가격이나 세일 정보 등등
일반적으로 변수로 발생하는 것들이에요

왜 그런문제가 발생하는 걸까요?

아래글에서 설명드렸지만 ai는 기본적으로 
편향성을 가지고있습니다 말은어렵지만

기본적으로 원래 학습된 정보를 따라가려는 성질이에요 

그러니까 그런 편향성에서 벗어나는 정보는
예외는 빈도가 낮아서 데이터에 적기에 오류발생률이 높아지죠

문제는 사용자는 그정보를 알고있기에
수정하라 반복시키고 이러한 곳에서 시간낭비가 심해

체감 피해가 크다는 악순환이 반복됩니다.

오늘의 글은 몇백명일떄부터 친했던 스레더중 
@bocchi.ai 님의 이야기중 이 롱테일 문제에대한 이야기기도 해요

사실 테일리스크 롱테일은같은 이키워드는
이쪽 분야에서는 꽤나 자주쓰이는 용어로

롱테일 문제
"클래스 불균형이 심한 현실 데이터가 있는경우”
말이어렵죠?

쉽게말하면 
다수케이스에는 잘맞는데 소규모 변수가 데이터가 적다
그러니까 일반적으론 맞는데 특정부분에서 틀린정보가있다.

여기에 테일리스크는 
”드물게 터지지만 한 번 터지면 손해가 큰 실패를 위험으로 관리해야함”

이런느낌으로 사용되는 용어로 
사용자 측면에서 고치기위해 노력이나 시간이 많이 투입되는 문제입니다.

이론적으론  클래스 불균형, 데이터 분포 편향, 희귀 사례/저빈도 사건, OOD 일반화 등등

지루하고 현학적인 설명이 들어가지만 이걸굳이 아실필요는없어요

다만 우리가 알아야하는건 그래서 이러한 문제를 어떻게 대응하냐입니다.

언어모델은 대체로 많은데이터중 익숙한 대답 그리고 
어떻게든 결론을 내는 방향으로 최적화됩니다.

Mitigating Frequency Bias and Anisotropy in Language Model Pre-Training with Syntactic Smoothing 같은연구에서 다루는주제로
출력이 자꾸 가장 흔한 전제와 가장 흔한 결론을 낸다는 거죠

그러니까 소위말하는 볼만한데
알맹이없는 글이 탄생하는다는거에요

그래서 문제는 이걸 고치기위해
이런현상이 있어를 중간에 추가해줘도

평균 회귀(regression to the mean)같은 현상이있기에,

당장은 반영을하지만 대답이 좀만길어지면 
다시 원래응답으로 돌아가는겁니다.
