# Supabase는 항상 좋은 선택지일까

Supabase는 항상 좋은 선택지일까?

우선 나는 supabas를 찬양하다 못해 적극 사용하고 있는 개발자야. 이미 프로덕션 레벨에서 많이 쓰고 있고 주변에 추천도 정말 많이 해왔어. 

나만의 소중한 인디밴드같은 느낌이던 녀석이 이제 대부분 아는 프로덕트가 된게 뭔가 성장드라마 보는 것 같아서 좋긴한데 그러다보니 또 무분별한 추천이 많은 것 같아서 크리티컬한 단점들, 그리고 쓰면 오히려 역효과가 날 수 있는 사람들을 적어볼게!

1. 백엔드 지식이 없는 입문자에겐 오히려 역효과가 날 수 있음
 : 오히려 백엔드를 어느정도 다룰줄 알고, db설계나 스테이징, 운영환경 분리, 디비 백업 등 여러 지식이 이미 있는 경우엔 정말 trade off를 고려해서 빠른 속도로 서비스 런칭할 수 있다는 점에서 백엔드 숙련자에겐 정말 좋은 서비스야. 다만 백엔드 지식이 없는 사람이 썼을땐 실제 운영환경에서 어려움을 겪을 확률이 아주 커.

2. edge function이 Deno라서 조금 귀찮은 점들이 있음
 : supabase는 아키텍쳐 최적화를 위해서 적용했는지 왜 저걸 썼는지 나는 정확히 모르지만 어찌됐건 조금 Minor한 기술인 Deno 환경을 쓰고 있어서 dependency 등에서 어려움을 겪을 확률이 있어. 사실 무료로 이정도로 쓸 수 있는게 어딘가 싶긴 한데 여튼 조금 불편한 단점이긴 해. 

그래서 나는 supabase를 쓰더라도 edge function은 google cloud run이나 aws lambda를 사용하는게 더 편할때도 있어. 이것들도 무료 플랜이 넉넉하거든

(계속)

3. 제일 치명적인 단점인데 transaction 처리를 하려면 코드 구조가 상당히 복잡해짐 
 : 라이브 프로덕션에서는 사실 transaction도 중요한 요소중에 하나인데 supabase는 기본적으로 클라이언트에서 RLS로 db조작을 하기 때문에 client만으로는 db transaction이 불가능해. 

하려면 RPC를 사용해야하고, postgresql에 함수 형태로 등록하는 것이기 때문에 코드 유지관리가 진짜 빡세져. 

자주 수정이 필요한 함수면 코드 형태로 어디 존재하는게 아니고 이걸 create or replace로 계속 Migration을 해줘야하는거야.  migration 파일에만 코드가 존재하는거지.. 

또한 클라이언트에서 제공하지 않은 조회나 insert, join을 하려면 마찬가지로 RPC를 써야하고 이게 코드 유지관리에 진짜 치명적으로 안좋음
