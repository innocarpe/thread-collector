# 이 Trigger의 문제점은 디버깅과 유지보수를 지옥으로 빠뜨리게 돼

이 Trigger의 문제점은 디버깅과 유지보수를 지옥으로 빠뜨리게 돼. 특히 여러 개의 Trigger가 연쇄적으로 일어나는 경우 멘탈 나갈 수 있지. 실행 흐름 자체를 추적하기 어려워져. 

mvp단계에서는 일단은 상관없겠지만 데이터가 꽤 많아지고 trigger도 많으면 Lock경합이 증가하면서 엄청난 트랜잭션 성능저하가 생기기도 해

결론은 진짜 필요한 상황 아니면 쓰면 안되는 기능이고 현업에서 쓰면 쌍욕먹을 수 있는 기능인데 AI 코딩 툴은 시킨거 일단 하긴 해야되니까 이걸 슥 권하는 경우가 진짜 많아. 내가 AI가 만든 결과를 보고 두 눈을 의심한 경우가 많음.. 심지어 나는 현재 구조의 한계를 인지하고 trigger를 안써도 되는 요청사항을 준건데도 얘가 그냥 쓴거야.

RLS와 마찬가지로 Trigger 또한 db 테이블에 쿼리 코드가 저장돼. 
비즈니스 로직이 애플리케이션 코드가 아닌 데이터베이스 레벨에 위치하게 되어 버전 관리와 배포가 너무 어려워져.
