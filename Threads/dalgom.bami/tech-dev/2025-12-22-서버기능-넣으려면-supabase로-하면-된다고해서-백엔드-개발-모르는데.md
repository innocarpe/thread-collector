# 서버기능 넣으려면 supabase로 하면 된다고해서 백엔드 개발 모르는데 서비스 일단 어찌어찌 바이브코딩으로 만들었고 어찌됐건 유저 생겼는데 기...

서버기능 넣으려면 supabase로 하면 된다고해서 백엔드 개발 모르는데 서비스 일단 어찌어찌 바이브코딩으로 만들었고 어찌됐건 유저 생겼는데 기능업데이트하다가 디비 데이터 날아가면 어떡하지?에 대해서 답변 당장 못하는 사람은 이 글을 꼭 저장해두고 바로 실천하세요.

docker로 local supabase띄우기, migration file 관리, staging supabase 따로 배치, supabase db push, supabase migrations up..

이 명령어들 뭔지 모르겠고 써본적 없으면 일단 아래처럼이라도 하세요. 그래야 나중에 개발자들이 여러분들 데이터 살려줄 수 있습니다. 언젠간 저한테 고마워 할겁니다..(계속)

1. 가장 쉬운 방법: 최소한 supabase pro플랜을 쓰세요, 자동으로 최근 7일에 대해서 db백업이 매일 돌아갑니다. 뭔가 데이터 날아가고 망했다 싶을때 백엔드 개발자를 수소문해서 구해서 supabase에서 백업은 해줬다는데.. 어떻게 살려줄 수 있어? 라고 물어봅니다.

2. 저거보단 어려운데 돈 안드는 방법:

우선 supabase client를 설치합니다. (https://supabase.com/docs/guides/local-development)

보통은 가이드대로 개발자라면 supabase link를 통해 프로젝트를 연결하지만 여러분들은 이상한 명령어 모르고 날려서 순식간에 나락으로 갈 수 있습니다. 정확하게 이게 뭐하는 건지 인지하기 전엔 절대 연결하지마세요. 참고로 클로드코드나 이런놈들 퍽하면 오 깔끔하게 supabase db reset 이거 한번 돌려보죠? 이러면서 제안하는데 이게 나락 지름길입니다. 데이터 다 날아감 저거 돌리면(계속)

설치한 다음 이 명령어를 치면 backup.sql이라는 파일이 pc에 생길거에요. 이게 바로 여러분들 db를 전체 백업한 생명줄 파일입니다. 이걸 매일 실행해서 파일명 오늘날짜로 바꾸고 차곡차곡 쌓으세요. 그럼 위기때 마찬가지로 개발자 헬프 요청 가능합니다

[] 부분은 여러분이 채워주셔야해요. db비번(계정비번 아니고 db비번입니다)은 supabase가입할때 만들었을거고 project_ref는 세팅들어가면 project_id라고 있을거에요.

supabase db dump --db-url "postgresql://postgres:[PASSWORD]@db.[PROJECT_REF].supabase.co:5432/postgres" -f backup.sql

ex)supabase db dump --db-url "postgresql://postgres:123412341234@db.dkjuansgfsyvp.supabase.co:5432/postgres" -f backup.sql

스레드가 이상한곳에 링크걸어서 복붙이 잘 안되는데 ex부분꺼 보고 자기 값으로 바꿔서 하시면 됩니다.. 그냥 맘편하게 supabase pro쓰는게 나을 수도 있습니다.

실제 사용자 데이터가 중요한 서비스를 한다면 꼭 백엔드 기초 지식을 쌓기 시작하세요. 없이 운영, 유지보수한다는건 말이 안됩니다. 10명내외로 쓰면 모를까
