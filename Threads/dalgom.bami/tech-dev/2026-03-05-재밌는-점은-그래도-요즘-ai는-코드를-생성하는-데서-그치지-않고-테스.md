# 재밌는 점은 그래도 요즘 AI는 코드를 생성하는 데서 그치지 않고, 테스트 코드를 직접 실행해보고 로그를 보면서 잠재적인 버그를 스스로 인식하고...

재밌는 점은 그래도 요즘 AI는 코드를 생성하는 데서 그치지 않고, 테스트 코드를 직접 실행해보고 로그를 보면서 잠재적인 버그를 스스로 인식하고 수정하기도 합니다. 최근 에이전트로서의 능력이 폭발적으로 성장했기때문입니다. 테스트 코드가 잘 갖춰진 환경일수록 AI도 더 정확하게 동작합니다. 사람만을 위한 안전망이 아니라 AI가 더 잘 작동하기 위한 환경이기도 한 셈입니다.

사실 이 구조의 중요성은 이미 오픈소스 세계에서 증명이 됐습니다. 규모가 큰 오픈소스 프로젝트들은 전 세계 수많은 컨트리뷰터의 코드를 안전하게 받아내기 위해 테스트 자동화가 잘 구성되어 있습니다. SQLite가 대표적인 사례입니다. 소스코드는 공개하면서도 핵심 테스트 suite인 TH3는 공개하지 않습니다. 소스 코드의 절반 이상이 테스트 관련 코드일 만큼 검증에 공을 들이고 있고, 그 핵심 부분만큼은 경쟁력으로 보고 지키고 있는 것입니다.

최근 Cloudflare의 엔지니어 한 명이 AI로 next.js를 vite기반으로 1주일만에 아예 다시만든 vinext도 next.js의 테스트케이스가 정말 풍부하게 잘 구성되어 있어서 가능했다고 합니다.

결국 AI 코딩툴의 진짜 수혜자는 테스트 자동화가 잘 된 팀이 아닐까 생각합니다. 검증 시스템이 촘촘할수록 AI가 생성한 코드를 더 빠르고 담대하게 반영할 수 있습니다. 이 시스템이 없는 팀은 초반엔 속도가 나다가, 서비스가 커지는 순간 배포가 무서워집니다. AI를 쓰면서도 점점 느려지는 상황이 오게 됩니다.

이제 개발팀의 실력은 코드 생성 속도로 측정되지 않을 것 같습니다. 얼마나 빠르고 안전하게 배포할 수 있는가, 그 신뢰 구조를 얼마나 잘 만들었는가로 갈리지 않을까 싶습니다. CI/CD, 운영의 복잡한 케이스들을 대부분 포함하는 테스트 커버리지, 스테이징 환경. 화려하지 않지만 이게 AI 시대의 진짜 경쟁력이 아닐까 생각합니다.

AI가 코드를 짜주는 시대, 정작 병목은 다른 곳에 생겼습니다.

배포 버튼 앞에서 손이 멈춥니다. 이거 올려도 되나? 이 질문 앞에서 개발자는 AI가 절약해준 시간을 그대로 다시 씁니다. 코드 생성에 5분, 검토에 50분. 거침없던 프로젝트 초반과 다르게 점점 신경쓸게 늘어나고 머뭇거리게 됩니다.

AI가 만든 코드가 나쁜건 아닙니다. 80점정도 수준까진 아주 잘 짭니다. 근데 그럴듯하다는 게 가장 큰 문제입니다. 아예 틀린 코드는 눈에 금방 잡힙니다. 80점짜리 코드는 돌아가긴 하는데 우리 시스템 맥락에서 위험할 수도 있습니다. 눈으로 봐선 모릅니다. 리뷰도 통과합니다. 그리고 프로덕션에서 갑자기 문제가 터집니다.
