# 예를들어 user_id =3 이 작성한 글에 user_id=4가 좋아요를 눌러서 좋아요를 +1 해야되는 상황

예를들어 user_id =3 이 작성한 글에 user_id=4가 좋아요를 눌러서 좋아요를 +1 해야되는 상황 같이 여러 사용자가 엮여 있는 경우도 꽤 많고, 이것도 RLS를 더 복잡하고 잘 짜면 해결이 가능하지만 이거는 AI 코딩툴이 "이랏샤이마세!" 하면서 권한을 다 풀어버리는 경우가 엄청 많았음. 이렇게 되면 특정 악의적인 클라이언트가 내 db를 오염시키는게 가능해지고 이게 보안 구멍이 되는 것.

즉 AI 코딩툴에 요청할 때 명확한 제약사항을 백엔드 지식을 동반해서 전달해야 RLS가 잘 세팅될까 말까이고 RLS는 ai가 많이 학습했을 일반적인 코딩 방식이 아니기 때문에 더 심각한 문제가 생길 수 있음.

오늘 스타트업 서비스 하나 들어갔는데 마침 supabase길래 network tab에서 fetch 요청들을 둘러보다가 경악할만한 response들을 많이 봄.. 난 비로그인 사용자인데 다른 사람의 개인정보가 화면에만 안나올 뿐 response에 다 노출되어있었음
