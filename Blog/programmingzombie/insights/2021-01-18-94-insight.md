## 핵심 주장 (≤5)
1. 안드로이드에서 푸시 알람을 처리하는 일반적인 방식으로 FCM이 사용된다.
2. 푸시 구현 자체보다 내가 만든 코드가 잘 동작하는지 검증하는 테스트 방법 정리가 중요하다.
3. Firebase 콘솔은 가장 간단하지만 전송 가능한 항목이 제한적이라 간단한 확인용 테스트에 적합하다.
4. 레거시 FCM HTTP 프로토콜은 시작은 쉬울 수 있지만 플랫폼별 분기와 확장 측면에서 번거로울 수 있다.
5. Firebase는 보안, 플랫폼 맞춤설정, 확장성 측면에서 FCM HTTP v1 사용을 권장한다.

## 언급된 도구·서비스
- Firebase Cloud Messaging(FCM): 안드로이드 푸시 알람 처리와 테스트에 사용하는 메시징 서비스
- Firebase 콘솔: 프로젝트에서 Cloud Messaging 알림을 생성하고 간단한 푸시 테스트를 보내는 도구
- Postman: 레거시 FCM HTTP API와 HTTP v1 API를 직접 호출해 푸시 메시지를 보내는 도구
- Terminal: `curl`로 FCM 엔드포인트를 호출해 푸시 메시지를 보내는 실행 환경
- Legacy FCM HTTP 프로토콜: 서버 키 기반으로 푸시를 발송하고 기기 그룹 관리에 쓰는 기존 API
- FCM HTTP v1 API: Firebase가 권장하는 최신 푸시 전송 API
- OAuth 2.0 Playground: Part2에서 최신 HTTP v1 사용법을 설명할 때 참조한 도구

## 복제 가능한 전술 (≤3)
1. Firebase 콘솔을 먼저 사용해 알림 도달 여부와 기본 데이터 수신만 빠르게 검증한다.
   - 구체적 스텝: Firebase 콘솔에서 프로젝트를 선택하고 `Cloud Messaging`으로 이동한 뒤 `새 알림`에서 제목과 텍스트를 입력하고, 타겟을 앱 또는 주제로 설정한 후 필요 시 추가 옵션에 데이터를 넣어 발송한다.
   - 예상 리소스: Firebase 프로젝트, 테스트 대상 앱, Firebase 콘솔 접근 권한
   - 예상 효과: 글에서 설명한 것처럼 "간단하게 알람이 가는지 데이터가 잘 받아지는지 정도만 테스트"할 수 있다.
2. Postman으로 레거시 FCM HTTP 요청을 만들어 디바이스 토큰이나 토픽 대상으로 푸시를 수동 테스트한다.
   - 구체적 스텝: Firebase 콘솔의 프로젝트 설정 `클라우트 메세징`에서 서버 키를 복사하고, Postman에서 `POST https://fcm.googleapis.com/fcm/send` 요청을 만든 뒤 `Authorization: key=<server_key>`와 `Content-Type: application/json` 헤더를 넣고, Body에 `to`와 `notification` 또는 `data` JSON을 작성해 전송한다.
   - 예상 리소스: Firebase 서버 키, Postman, FCM 토큰 또는 토픽명
   - 예상 효과: 콘솔 없이도 포스트맨을 통해 푸시 페이로드를 반복 검증할 수 있다.
3. 터미널 `curl` 요청으로 커스텀 `data` 페이로드를 보내 후속 처리 로직을 점검한다.
   - 구체적 스텝: `curl --request POST 'https://fcm.googleapis.com/fcm/send'` 요청에 인증 헤더와 JSON 바디를 넣고, `data` 객체 안에 `body`, `title`, 추가 키를 담아 전송한 뒤 앱에서 `remoteMessage` 내부 `data`를 받아 특정 페이지 이동이나 상세 API 호출에 필요한 값을 처리한다.
   - 예상 리소스: 터미널, Firebase 서버 키, 테스트용 FCM 토큰 또는 토픽
   - 예상 효과: 글에서 설명한 것처럼 푸시 클릭 이후 필요한 값을 앱으로 전달해 후속 동작을 검증할 수 있다.

## 원문 요약 (≤5문장)
글은 FCM 푸시 알림을 테스트하는 세 가지 방법으로 Firebase 콘솔, 레거시 FCM HTTP 프로토콜, FCM HTTP v1 API를 소개한다. 저자는 간단한 확인에는 Firebase 콘솔이 적합하지만, 더 직접적인 테스트를 위해 Postman과 터미널에서 레거시 FCM HTTP 요청을 보내는 방법을 예시와 함께 설명한다. 또한 `notification`과 `data` 페이로드의 차이를 구분해, 푸시 표시용 정보와 후속 처리용 데이터를 나눠 사용할 수 있다고 설명한다. 플랫폼별 맞춤설정과 보안, 확장성을 이유로 Firebase가 HTTP v1 사용을 권장한다는 점도 정리한다. 마지막으로 다음 편에서 최신 HTTP v1 방식 사용법을 다룰 것이라고 예고한다.
