## 핵심 주장 (≤5)
1. 회사 프로젝트는 끝없는 버그 픽스와 요구사항 반영을 반복하며 품질을 높여야 안정적으로 완료할 수 있다.
2. 자기계발서는 일회성 동기부여보다 꾸준한 자극제로 활용할 때 습관 형성에 도움이 된다.
3. TDD, 리팩토링, 클린코드 교육을 통해 빠른 구현 중심 학습이 아니라 설계, 테스트, 가독성이 중요한 학습임을 깨달았다.
4. 토이 프로젝트는 서비스 전체를 끝까지 돌려보는 경험과 새로운 기술 검증 기회를 주며, 아무것도 하지 않는 것보다 훨씬 낫다.
5. 좋은 품질의 서비스가 곧 좋은 서비스는 아니며, 고객이 원하거나 필요로 하는 서비스여야 성과가 난다.

## 공개된 숫자·지표
- 앱 안정성: Crash free 100% (기간: 회사 프로젝트 운영 시점, 출처: "정말 치열하게 버그 잡기위해 노력한 결과 내가 맡은 앱 Crash free 100%를 달성 하였다.")
- 무버그·무요구사항 운영 기간: 2개월째 (기간: 프로젝트 잠정 완료 시점 기준, 출처: "1년동안 열심히 작업한 프로젝트는 현재 2개월째 버그도 없고, 요구사항도 없어서 모두가 행복하게 사용하는 것으로 잠정 완료되었다.")
- 자기계발서 독서량: 10권 이상 (기간: 2018년, 출처: "자기 개발서 10권 이상 읽기")
- To do 기록 지속 기간: 3개월째 (기간: 기록 시점 기준, 출처: "To do 앱을 적극 활용하고 있는데, 어느덧 3개월째 꾸준히 To do 기록을 진행중에 있다.")
- 교육 기간: 5주간 (기간: TDD, Refactoring, Clean Code 교육, 출처: "5주간의 교육이 끝났다.")
- 프로젝트 전체 코드 재작성 횟수: 3번 (기간: 교육 미션 수행 중, 출처: "설계 자체를 잘못해서 프로젝트 전체 코드를 3번이나 갈아엎는 상황도 있었다.")
- 앱 개발 및 배포 수: 21개 (기간: 1년 동안, 출처: "정확히는 12/30일날 앱하나를 추가로 배포하여 21개가 되었다.")
- 다운로드 1만명 이상 앱 수: 3개 (기간: 1년 동안 배포한 앱 기준, 출처: "- 1만명 다운로드 앱 3개")
- 다운로드 5000명 앱 수: 3개 (기간: 1년 동안 배포한 앱 기준, 출처: "- 5000명 다운로드 3개")
- 체중 증가: 10kg 증가 (기간: 회사 다닌 1년 6개월 가량, 출처: "회사 다니는 1년 6개월 가량 동안 10kg이 졌다")
- 최근 1년 체중 증가: 7키로 (기간: 작년 대비, 출처: "몸무게는 작년과 비교하면 무려 7키로가 쪘다.")
- 목표 감량: 10Kg (기간: 내년 목표, 출처: "내년에는 10Kg 감량에 성공하는게 목표이다.")
- 평균 수면 시간: 하루 4시간 정도 (기간: 당시 생활 습관, 출처: "하루 평균 4시간 정도 잤는데, 원하는 결과물은 나왔지만 몸이 망가지는게 점점 보이기 시작했다.")
- 롤모델의 앱 제작 수: 700개 이상 (기간: 소개 시점 기준, 출처: "그리고 현재는 앱을 700개 이상 만드신 분이다.")
- 롤모델의 수익 증가 배수: 30배 (기간: 작년 발표 대비 현재, 출처: "아마 작년 발표 때 보다 현재는 수익이 30배는 증가하셔서")

## 언급된 도구·서비스
- To do 앱: 매일 한 일을 체크하고 위젯과 알림으로 습관을 유지하는 데 활용
- PHP: 사내 API 서버 언어로 사용하며 API 작성과 리팩토링에 활용
- SQL: CRUD 쿼리 작성과 쿼리 튜닝에 활용
- Kotlin: 토이 프로젝트의 앱을 100% Kotlin으로 구현하고, 이후 서버 개발 스택으로도 언급
- Spring Boot: 토이 프로젝트에서 API 구축에 활용
- AWS: 인프라 구축에 활용
- EC2: AWS 인프라 구성 요소로 사용
- S3: AWS 인프라 구성 요소로 사용
- RDS: AWS 인프라 구성 요소로 사용
- CodeDeploy: AWS 배포 구성 요소로 사용
- Android MVVM: 앱 구조 구현 방식으로 사용
- Koin Injection: Android 앱 의존성 주입에 사용
- Firebase: 여러 제품을 폭넓게 사용
- Firebase 크래시 및 레포트: 크래시 분석과 리포트 확인에 사용

## 언급된 다른 creator·앱
- 동욱이 형: 열심히 하면 스타급 개발자가 될 수 있음을 보여준 롤모델로 언급
- 케빈님: 개발자로서 돈으로부터 해방될 수 있음을 일깨워준 롤모델로 언급

## 복제 가능한 전술 (≤3)
1. 사용자와 실무자 의견을 주기적으로 수집해 서비스 품질 개선 루프를 만든다.
   - 구체적 스텝: 현재 앱 사용자나 실무자를 직접 만나 불편한 점과 필요한 기능을 묻고, 수집한 의견을 정리해 주기적으로 앱에 반영한 뒤 다시 운영 상태를 확인한다.
   - 예상 리소스: 사용자 인터뷰 시간, 앱 수정 배포 시간, 의견 정리 도구
   - 예상 효과: 버그와 요구사항이 줄어들고 모두가 행복하게 사용하는 상태에 가까워질 수 있다.
2. 자기계발서와 To do 앱을 결합해 습관 유지용 자극 시스템을 만든다.
   - 구체적 스텝: 출퇴근 시간처럼 고정된 독서 시간을 정하고, 읽은 날마다 To do 앱에 체크하며, 지칠 때는 새로운 책을 읽어 다시 자극을 공급하는 방식으로 반복한다.
   - 예상 리소스: 책, 출퇴근 시간, To do 앱
   - 예상 효과: 작심삼일성 실행을 더 오래 유지하고 책 읽기를 습관으로 만들 수 있다.
3. 토이 프로젝트를 끝까지 배포해 기술 학습과 시장 검증을 동시에 수행한다.
   - 구체적 스텝: 생활 속 불편이나 취미 관련 문제를 주제로 앱을 만들고, 앱과 필요한 API 및 인프라를 직접 구성해 배포한 뒤, 다운로드 수와 사용자 피드백을 보며 다음 개선이나 새 프로젝트에 반영한다.
   - 예상 리소스: Kotlin, Spring Boot, AWS, Firebase, 개발 및 배포 시간
   - 예상 효과: 서비스 전체 흐름을 경험하고 새로운 기술을 검증하며, 실제 사용자 피드백과 수입 가능성까지 확인할 수 있다.

## 원문 요약 (≤5문장)
글쓴이는 2018년에 회사 프로젝트를 안정화해 높은 품질을 달성했고, 실무자 의견 수렴을 반복하며 서비스를 잠정 완료 상태로 만들었다. 자기계발서 독서와 To do 앱 기록으로 꾸준함을 유지했고, TDD·리팩토링·클린코드 교육을 통해 구현 중심에서 설계와 테스트 중심으로 사고를 전환했다. 1년 동안 21개의 앱을 개발·배포하며 다양한 기술과 운영 경험을 쌓았고, 토이 프로젝트의 장단점과 시장성의 중요성을 체감했다. 연말에는 서버 개발 중심으로 커리어를 옮기며 더 큰 책임감을 다짐했다. 동시에 체력 관리와 수면 부족 문제를 반성하며 다음 해 개선 목표를 세웠다.

## 본문 포인트별 발췌
> 정말 치열하게 버그 잡기위해 노력한 결과 내가 맡은 앱 Crash free 100%를 달성 하였다.
> 그럼 작심 열흘이 신기하게도 계속 유지(?)된다.
> 그동안 내가 해왔던 학습은 새로운 기술이나 라이브러리 사용하는 방법에만 치중해왔거나, 우선 빠르게 문제 해결을 하는것에만 집중하고 개발이 끝나버리는 과정의 반복이었다.
> 확실히 아무것도 안하는것 보다는 훠얼씬 낫다!
> 결국 좋은 서비스란 고객이 원하거나 필요로 하는 서비스라는 걸로 결론 짓게되었다.
