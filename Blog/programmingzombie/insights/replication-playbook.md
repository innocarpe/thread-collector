# programmingzombie 전략 복제 플레이북

## 복제할 핵심 전략 (Top 3)
- 문제 발생형 마이크로 포스트: creator는 개인 개발과 운영 중 실제로 막힌 문제를 바로 글로 전환했다. TemperStone도 내부에서 막힌 이슈를 `재현 가능한 해결 문서`로 바꾸면 검색 유입, 팀 내부 재사용, 실무 신뢰를 함께 만들 수 있다.
- 재현 가능한 실행 기록: creator의 강점은 개념 설명보다 권한, 설정 파일, 명령, 테스트, 검증 절차까지 남긴 점이다. TemperStone도 환경 정보와 검증 루틴을 남겨야 같은 문제가 다시 왔을 때 문서가 바로 자산이 된다.
- 학습 큐레이션 + 문제 운영 체계: creator는 공식 문서와 좋은 글을 짧게 큐레이션하고, 문제를 보드에서 관리하며, 결국 "문제를 소프트웨어적으로 해결하는 사람"이라는 관점까지 확장했다. TemperStone도 콘텐츠 생산을 별도 마케팅 활동이 아니라 문제 운영 체계의 부산물로 설계해야 한다.

## Day 1 (오늘 당장)
- [ ] 최근 TemperStone 작업에서 실제로 막혔던 문제 1개를 고르고, `문제 상황 / 재현 조건 / 실제 해결 코드 또는 설정 / 검증 방법 / 참고 링크` 템플릿으로 초안을 만든다.
- 필요 도구: 이슈 보드, Markdown 템플릿, 로그/터미널, 코드 저장소, 공식 문서 링크 저장소
- 예상 소요 시간: 반나절
- [ ] `문제-리서치-개발-테스트-완료` 5단계 보드를 만들고, 현재 진행 중인 이슈를 모두 이 단계 중 하나에 넣는다.
- 필요 도구: Trello 또는 동등한 칸반 도구, 팀 공용 운영 규칙 문서
- 예상 소요 시간: 짧은 세팅 시간 + 팀 합의 시간
- [ ] 자주 잊는 설정값을 치트시트로 분리한다. 예: 권한, 환경변수, DB 시간대, 테스트 실행 절차, 배포/디버깅 명령.
- 필요 도구: 내부 위키 또는 Markdown 문서, 기존 설정 파일, 배포/테스트 로그
- 예상 소요 시간: 짧은 정리 시간

## Week 1 (첫 주)
- [ ] 월요일에 해결할 문제 카드를 잡고, 금요일까지 `문제-리서치-개발-테스트-완료` 흐름을 한 번 끝까지 돌린다.
- 우선순위 근거: creator는 문제를 발견 즉시 기록하고, 중복 문제는 Vote로 우선순위를 정했으며, 월요일 할당-금요일 진행 리듬으로 운영했다.
- 예상 결과: 누락되지 않은 문제 백로그, 상태가 보이는 칸반 보드, 이번 주에 완료된 카드와 테스트 흔적
- [ ] 첫 해결형 글을 게시한다. 글에는 환경 정보, 필요한 권한/설정, 재현 단계, 수정 포인트, 검증 결과를 반드시 포함한다.
- 우선순위 근거: creator의 글은 대부분 "막힌 문제를 바로 해결한 메모"였고, 특히 설정 누락이나 UI 상태 문제 같은 실제 장애를 그대로 문서화했다.
- 예상 결과: 내부 재사용이 가능한 첫 플레이북형 글, 동일 이슈 재발 시 바로 참조 가능한 문서
- [ ] TemperStone 팀이 반복해서 찾는 공식 문서·참고 블로그 링크를 큐레이션한다.
- 우선순위 근거: creator는 설명을 과하게 늘리기보다 MDN, 안드로이드 블로그, 예제 모음, Stack Overflow 같은 외부 자료를 빠르게 연결해 학습 진입 비용을 낮췄다.
- 예상 결과: 빠르게 참조 가능한 학습 링크 묶음, 문제 해결 속도를 높이는 공용 소스 라이브러리
- [ ] 최소 1개 핵심 흐름에 로컬 테스트 또는 검증 루틴을 붙인다.
- 우선순위 근거: creator는 장치 의존 테스트보다 빠른 로컬 Junit 환경을 만들었고, DB와 네트워크도 먼저 확인하는 유틸과 눈으로 검증하는 절차를 남겼다.
- 예상 결과: 코드 변경 후 바로 확인 가능한 테스트 또는 수동 검증 체크리스트

## Month 1 (첫 달)
- [ ] 제품 개발과 콘텐츠 생산을 같은 운영 체계에 묶는다. 이번 달 동안 해결한 문제를 모아 `TemperStone이 실제로 해결한 문제 아카이브`를 만든다.
- 마일스톤: 월요일 문제 선정, 금요일 상태 점검, 월말 또는 월초에 완료 카드 중 재사용 가치가 높은 항목을 글이나 문서로 승격
- 성공 지표: 문제 카드가 `문제` 단계에서 끝나지 않고 `테스트`와 `완료`까지 이동하며, 완료 카드 중 일부가 검색 가능한 문서로 남아 있음
- [ ] 자주 터지는 영역을 표준화한다. 예: 네트워크 상태 체크, 권한/설정 치트시트, 시간대 처리, 리스트 상태 관리, 배포·디버깅 명령.
- 마일스톤: 동일 유형의 이슈가 재발할 때 기존 치트시트나 유틸을 재사용
- 성공 지표: 새 이슈 대응 시 기존 문서·유틸·명령을 참조하는 사례가 생기고, 설정 누락 때문에 다시 막히는 일이 줄어듦
- [ ] TemperStone의 강점을 한 문장으로 정의하고, 이후 글의 제목과 도입부를 그 강점에 맞춘다.
- 마일스톤: "우리는 무엇을 잘하는 팀인가"에 대한 내부 합의 문장 확정
- 성공 지표: 글이 단순 작업 일지보다 "이 팀은 실제 문제를 끝까지 푸는 팀"이라는 인상을 일관되게 전달함

## 주의사항 & 함정
- 실제로 겪지 않은 문제를 억지로 일반론으로 쓰면 creator의 장점이 사라진다. 이 creator의 글은 대부분 개인 개발 중 막힌 지점을 바로 해결한 기록이었다.
- 설정·권한·환경 정보를 빼면 재현성이 무너진다. 소켓 권한, 네트워크 퍼미션, SSH 설정, SQLite 시간대, 테스트 폴더 구조 같은 세부가 핵심이었다.
- UI나 증상만 고치고 상태를 데이터나 시스템 레벨에서 관리하지 않으면 같은 문제가 반복된다. RecyclerView 체크 상태, Fragment BackStack, 네트워크 분기, 트랜잭션 관련 글이 이 점을 보여준다.
- 외부 글을 복사해오는 수준에 머물면 안 된다. creator는 참고 링크를 많이 썼지만, 실제로 따라 해보고 "무엇이 동작했고 왜 필요한가"를 자기 문맥으로 다시 적었다.
- 인터럽트, 시간대, 스케줄링 지연처럼 눈에 잘 안 띄는 운영 이슈를 무시하면 나중에 비용이 커진다.
- TemperStone 컨텍스트에서는 안드로이드/자바 자체를 복제하는 것이 아니라, `자사 스택에서 발생한 실제 문제를 가장 짧고 검증 가능하게 푸는 방식`을 복제해야 한다.
- TemperStone이 성장하려면 코드 내부 최적화만이 아니라 운영 큐, 문서화, 우선순위, 팀의 강점 설명까지 문제 해결 범위를 넓혀야 한다.
