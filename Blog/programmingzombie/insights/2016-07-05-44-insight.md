## 핵심 주장 (≤5)
1. 자바의 동기화 영역은 `synchronized` 키워드로 표시되며, 경합 조건을 피하기 위한 방법으로 쓰인다.
2. 같은 객체를 기준으로 동기화된 블록은 한 시점에 오직 한 스레드만 실행할 수 있고, 다른 스레드는 블록이 끝날 때까지 대기한다.
3. 인스턴스 메소드 동기화는 해당 인스턴스를 기준으로, 스태틱 메소드 동기화는 클래스 객체를 기준으로 이루어진다.
4. 메소드 전체를 잠그지 않고 필요한 코드 블록만 `synchronized`로 감싸는 편이 더 효율적일 수 있다.
5. 서로 다른 객체를 모니터로 사용하면 각 동기화 블록이나 메소드는 동시에 실행될 수 있다.

## 공개된 숫자·지표
- `synchronized` 적용 유형: 4가지 (기간: 설명 구간, 출처: "synchronized 키워드는 다음 네 가지 유형의 블록에 쓰인다.")

## 복제 가능한 전술 (≤3)
1. 공유 상태를 바꾸는 임계 구역만 `synchronized(this)`로 감싸서 동기화 범위를 최소화한다.
   - 구체적 스텝: 공유 필드를 변경하는 메소드를 찾고, 메소드 전체를 잠글 필요가 없으면 실제 상태 변경 코드만 `synchronized(this){ ... }` 블록 안으로 옮긴다.
   - 예상 리소스: 자바 코드 수정, 공유 객체 식별 시간, `synchronized` 키워드
   - 예상 효과: 메소드 전체 대신 필요한 구간만 동기화해 같은 기능을 유지하면서 더 효율적으로 구성할 수 있다.
2. 여러 스레드가 같은 인스턴스를 공유하도록 설계해 한 번에 하나씩만 메소드가 실행되게 만든다.
   - 구체적 스텝: 공유 카운터 같은 객체를 하나 생성하고, 각 스레드 생성자에 동일한 인스턴스를 전달한 뒤 해당 객체의 `synchronized` 인스턴스 메소드를 호출하게 한다.
   - 예상 리소스: 공유 객체 1개, 스레드 클래스, `synchronized` 인스턴스 메소드
   - 예상 효과: 같은 인스턴스를 동기화 기준으로 사용하므로 한 시점에 한 스레드만 임계 메소드를 실행하게 된다.
3. 서로 독립적인 작업은 서로 다른 모니터 객체를 사용해 병렬 실행 가능하게 분리한다.
   - 구체적 스텝: 현재 같은 `this`나 같은 클래스 객체를 기준으로 잠그는 코드가 있는지 확인하고, 동시에 실행해도 되는 작업이면 별도의 객체를 동기화 괄호 안에 전달해 잠금 기준을 분리한다.
   - 예상 리소스: 별도 모니터 객체, 자바 동기화 블록 수정
   - 예상 효과: 동기화 기준이 달라져 서로 다른 메소드나 블록이 동시에 실행될 수 있다.

## 원문 요약 (≤5문장)
이 글은 자바의 `synchronized`가 무엇을 잠그는지 인스턴스 메소드, 스태틱 메소드, 코드 블록 단위로 나눠 설명한다.
핵심은 동기화가 "코드" 자체가 아니라 특정 객체나 클래스 객체를 기준으로 이루어진다는 점이다.
같은 모니터 객체를 공유하면 한 번에 한 스레드만 임계 구역을 실행할 수 있고, 다른 객체를 기준으로 잠그면 동시에 실행될 수 있다.
글 말미에는 `Counter` 예제를 통해 같은 인스턴스를 공유할 때와 서로 다른 인스턴스를 사용할 때의 차이를 보여준다.

## 본문 포인트별 발췌
> 자바 동기화 블록은 메소드나 블록 코드에 동기화 영역을 표시하며 자바에서 경합 조건을 피하기 위한 방법으로 쓰인다.
> 같은 객체에 대한 모든 동기화 블록은 한 시점에 오직 한 쓰레드만이 블록 안으로 접근하도록 - 실행하도록 - 한다.
> 동기화가 반드시 메소드 전체에 대해 이루어져야 하는 것은 아니다.
> 종종 메소드의 특정 부분에 대해서만 동기화하는 편이 효율적인 경우가 있다.
> 여기서 만일 두 쓰레드가 서로 다른 Counter 인스턴스를 전달받았다면, add() 메소드는 동시에 호출될 수 있을 것이다.
