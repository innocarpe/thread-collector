## 핵심 주장 (≤5)
1. AWS 계정 간 EC2 이전은 스냅샷 생성 후 이미지(AMI)로 만들고, 대상 계정에 이미지 권한을 부여하는 방식으로 진행할 수 있다.
2. EC2를 AMI 기반으로 복제하면 처음 인스턴스를 만들 때처럼 세세한 설정을 다시 하지 않아도 된다.
3. 복제 후에는 보안그룹이나 일부 커스텀 설정만 다시 손보면 기존과 유사한 인스턴스를 빠르게 만들 수 있다.
4. 작성자는 AWS 개인 개발 서버를 다른 AWS 계정으로 옮긴 경험을 EC2부터 순차적으로 정리하고 있다.

## 언급된 도구·서비스
- AWS: 개인 개발 서버를 다른 계정으로 이전하는 대상 플랫폼
- EC2: 스냅샷 생성, 이미지 생성, 권한 부여, 인스턴스 복제에 사용된 서비스
- RDS: 계정 이전 대상 서비스 예시로 언급
- S3: 계정 이전 대상 서비스 예시로 언급
- Route 53: 계정 이전 대상 서비스 예시로 언급
- CodeDeploy: 계정 이전 대상 서비스 예시로 언급
- 스냅샷: 기존 EC2 인스턴스 상태를 기반으로 복제 준비에 사용
- AMI: 스냅샷으로부터 생성되어 다른 계정에서 인스턴스를 띄우는 데 사용되는 이미지

## 복제 가능한 전술 (≤3)
1. AWS 계정 간 EC2를 AMI 권한 공유 방식으로 이전한다.
   - 구체적 스텝: 원본 계정에서 대상 인스턴스의 스냅샷을 생성하고, 해당 스냅샷으로 이미지를 생성한 뒤, 이미지 권한 수정에서 대상 AWS 계정 ID를 추가한다. 이후 대상 계정에서 `나의 AMI` 목록에서 해당 이미지를 선택해 새 인스턴스를 시작한다.
   - 예상 리소스: AWS 계정 2개, EC2 콘솔 접근 권한, 스냅샷/AMI 생성 작업 시간
   - 예상 효과: 기존 인스턴스를 다른 AWS 계정으로 비교적 간단하게 복제할 수 있다.
2. 복제 이후에는 차이 나는 설정만 최소 범위로 조정한다.
   - 구체적 스텝: AMI로 인스턴스를 띄운 뒤 기본 설정이 기존과 동일한지 확인하고, 보안그룹이나 특정 커스텀 항목만 다시 수정한다.
   - 예상 리소스: EC2 인스턴스 생성 권한, 보안그룹 설정 시간
   - 예상 효과: 신규 인스턴스를 처음부터 세세하게 다시 구성하는 부담을 줄일 수 있다.

## 원문 요약 (≤5문장)
작성자는 개인 개발 서버를 새로 구성하는 대신 AWS A 계정의 서버를 AWS B 계정으로 옮기는 방식으로 방향을 바꿨다. 이 글은 여러 AWS 서비스 중 EC2 이전 절차를 먼저 정리한 기록이다. 핵심 절차는 EC2 스냅샷 생성, 스냅샷 기반 이미지 생성, 대상 계정에 대한 이미지 권한 부여, 대상 계정에서 해당 AMI로 인스턴스 시작이다. 이렇게 복제하면 대부분의 설정이 따라오므로 보안그룹 등 일부만 조정하면 된다.

## 본문 포인트별 발췌
> 서버 인프라 공부도 할 겸 개인 개발 서버를 처음부터 다시 모두 설정 해보기로 했다.(하다가 너무 일이 커질것 같아서 옮겨보는 걸로 변경) A AWS 서버를 B AWS 서버로 모두 옮기는 작업을 해보자.
> 어떤 것들을 이전 해야할까 생각해보니 당연히 내가 사용하고 있는 AWS에 서비스들이었다. (EC2, RDS, S3, Route 53, CodeDeploy) 괄호 안에 있는 서비스 들을 A 계정에서 B 계정으로 옮겨본 경험을 기록하고자 한다.
> 스냅샷 생성이 완료되면 해당 **스냅샷을 선택 후 작업 > 이미지 생성을 클릭 > 스냅샷 뜬 이미지를 이미지 생성**하는 작업을 진행합니다.
> **생성된 이미지를 선택하고 작업 > 이미지 권한 수정을 클릭**하면 아래와 같은 팝업이 뜹니다.
> 왠만한 부분은 기존과 동일하게 설정이 되어있고, 보안그룹이나 특정 부분을 커스텀 하고싶은 부분만 다시 손보면 간단하게 인스턴스가 복제된 것을 확인 할 수 있습니다.
