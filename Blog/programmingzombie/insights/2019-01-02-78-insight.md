## 핵심 주장 (≤5)
1. `Failed to load ad: 3`는 광고 요청 자체의 실패가 아니라 광고 인벤토리가 없는 상태를 뜻한다.
2. 앱이 PlayStore에 배포된 이후에만 실제 광고가 노출될 수 있으므로, 개발 중에는 테스트 광고로 검증해야 한다.
3. 디버그와 릴리즈 환경에 따라 광고 ID를 자동으로 스위칭하면 개발 단계와 배포 단계 모두에서 광고 동작을 안정적으로 확인할 수 있다.

## 공개된 숫자·지표
- 광고 로드 에러 코드: 3 (기간: 해당 문제 발생 시점, 출처: `admob onAdFailedToLoad !! 3과 함께 W/Ads: Failed to load ad: 3의 에러 문구가 노출되는 것이 아닌가.`)

## 언급된 도구·서비스
- Google Mobile Ads SDK: `MobileAds.initialize`, `AdRequest`, `InterstitialAd`, `AdListener`를 사용해 배너/전면 광고를 구현하는 데 사용됨
- AdMob: 실제 광고 ID와 테스트 광고 ID를 구분해 광고 노출을 운영하는 서비스로 언급됨
- Google Document: 에러 코드 `ERROR_CODE_NO_FILL` 의미를 확인하는 참고 문서로 사용됨
- PlayStore: 실제 광고가 노출되는 배포 환경으로 언급됨

## 언급된 다른 creator·앱
- 없음

## 복제 가능한 전술 (≤3)
1. 디버그와 릴리즈 환경에 따라 광고 ID를 자동 전환한다.
   - 구체적 스텝: `BuildConfig.DEBUG`를 기준으로 테스트 광고 ID와 실제 광고 ID를 선택하는 헬퍼 메서드를 만든다. 배너와 전면 광고 초기화 시 `adUnitId`에 이 헬퍼를 적용한다. 개발 중에는 테스트 광고 노출 위치와 시점을 확인하고, 릴리즈 빌드에서는 실제 광고 ID가 적용되는지 검증한다.
   - 예상 리소스: Android 프로젝트 코드 수정, AdMob 테스트 광고 ID, 실제 광고 ID
   - 예상 효과: 개발 중에도 광고 UI와 타이밍을 점검할 수 있고, 릴리즈 후에는 실제 광고가 정상 노출되는 흐름을 분리해 운영할 수 있다.
2. 광고 로드 실패 코드가 나오면 먼저 공식 문서에서 의미를 확인한다.
   - 구체적 스텝: `onAdFailedToLoad` 콜백에서 에러 코드를 로그로 남긴다. 공식 문서에서 해당 코드 의미를 확인해 코드 문제인지 인벤토리 문제인지 구분한다. 인벤토리 부족이나 배포 조건 문제라면 구현을 바꾸기보다 테스트 광고 전략으로 전환한다.
   - 예상 리소스: 로그 확인, Google AdMob 공식 문서
   - 예상 효과: 구현 자체는 정상인데 광고가 안 나오는 상황을 빠르게 진단하고 불필요한 코드 수정 시간을 줄일 수 있다.

## 원문 요약 (≤5문장)
작성자는 Android에서 전면 광고를 구현했지만 개발 모드에서 `Failed to load ad: 3` 오류가 발생해 광고가 노출되지 않는 문제를 겪었다. 공식 문서를 확인한 결과 이 코드는 광고 요청은 정상적으로 갔지만 반환할 광고 인벤토리가 없다는 의미였다. 글에서는 현재 실제 광고는 앱이 PlayStore에 배포된 뒤에 노출된다고 설명한다. 그래서 개발 중에는 테스트 광고 ID를 사용하고, 릴리즈 시에는 실제 광고 ID를 쓰도록 자동 스위칭하는 방식을 적용했다. 그 결과 개발 중에는 테스트 광고로 위치와 타이밍을 확인하고, 릴리즈 후에는 실제 광고 노출을 확인할 수 있었다.

## 본문 포인트별 발췌
> admob onAdFailedToLoad !! 3과 함께
> W/Ads: Failed to load ad: 3의 에러 문구가 노출되는 것이 아닌가.
> 광고 요청은 정상적으로 전달되었지만 전달받은 광고 인벤토리가 없다는 뜻이었다.
> 예전에는 앱 배포전에 Debug 상태에서도 광고 출력이 잘되었지만, 현재는 앱이 PlayStore에 배포되고 난 이후에 광고가 노출된다고 한다.
> Test용 광고 ID or 실제 광고 ID현재 개발중인지 릴리즈인지 판단하여 자동으로 광고 ID가 스위칭 되도록 작업해주었다.
