## 핵심 주장 (≤5)
1. `Timer`는 같은 이름으로 객체를 여러 개 만들면 마지막에 생성한 객체만 `close()`로 종료할 수 있고, 나머지 실행 중인 객체들은 종료할 수 없다고 설명한다.
2. `ScheduledExecutorService`는 객체를 여러 개 만들어 실행시키더라도 `shutdownNow()`로 일괄 종료할 수 있다고 주장한다.
3. 작성자는 구글링과 Stack Overflow 참고 결과를 바탕으로 확장성 등 여러 측면에서 `ScheduledExecutorService`가 더 낫다고 본다.

## 언급된 도구·서비스
- ScheduleExecutorService: 일정 시간마다 메서드를 실행하는 예제로 소개되며, 여러 실행 객체를 `shutdownNow()`로 일괄 종료하는 용도로 설명됨
- Timer: 일정 시간마다 메서드를 실행하는 비교 예제로 제시되며, 여러 객체 생성 시 종료 제어에 한계가 있는 방식으로 설명됨
- Stack Overflow: `ScheduleExecutorService`가 더 낫다는 판단의 참고 출처로 언급됨

## 복제 가능한 전술 (≤3)
1. 주기 실행 작업이 여러 개일 때 `ScheduledExecutorService` 하나로 스케줄을 모아 관리한다.
   - 구체적 스텝: `Executors.newScheduledThreadPool(1)`로 스케줄러를 만든다 -> 작업별 `Runnable`을 정의한다 -> `scheduleAtFixedRate`로 각 작업의 주기를 등록한다 -> 종료 시 `isShutdown`과 `isTerminated` 상태를 확인한 뒤 `shutdownNow()`를 호출한다
   - 예상 리소스: Java 실행 환경, 주기 작업별 `Runnable` 정의 시간
   - 예상 효과: 여러 실행 객체를 한 번에 종료할 수 있어 종료 제어가 단순해진다
2. `Timer` 기반 구현을 유지해야 한다면, 여러 `Timer` 인스턴스를 계속 새로 만들지 말고 종료 제어 한계를 먼저 점검한다.
   - 구체적 스텝: 현재 코드가 `new Timer(...)`를 반복 생성하는지 확인한다 -> 어떤 인스턴스가 실제로 참조되고 종료되는지 추적한다 -> 다중 타이머를 개별 종료해야 한다면 구조를 `ScheduledExecutorService` 중심으로 재구성한다
   - 예상 리소스: 기존 스케줄 코드 검토 시간, Java 리팩터링 작업
   - 예상 효과: 마지막 객체만 종료되고 나머지가 계속 실행되는 문제를 피할 수 있다

## 원문 요약 (≤5문장)
이 글은 Java에서 일정 시간마다 메서드를 실행하는 두 방식인 `ScheduledExecutorService`와 `Timer`를 비교한다. 작성자는 `Timer`는 여러 객체를 만들 경우 마지막 객체만 종료할 수 있고 나머지는 종료하기 어렵다고 설명한다. 반면 `ScheduledExecutorService`는 여러 실행 객체를 돌리더라도 `shutdownNow()`로 일괄 종료할 수 있다고 본다. 그 근거로 각각의 예제 코드를 제시하고, Stack Overflow를 참고해 확장성 측면에서도 `ScheduledExecutorService`가 더 좋다고 정리한다.

## 본문 포인트별 발췌
> Timer는 같은 이름으로 객체를 여러개 만들었을 경우 가장 마지막에 생성한
> 객체는 close()로 종료 할 수 있으나, 나머지 실행중인 객체들은 종료할 수 없다.
> 반면, ScheduleExecutorService의 경우 객체를 여러개 만들어 실행시키더라도 shutdownNow()를 사용하여 일괄종료 할 수 있다.
> 구글링 결과 Stack Overflow의 내용을 참조한 결과 확장성이나 여러가지로 ScheduleExecutorService가
> 좋다고 한다.
