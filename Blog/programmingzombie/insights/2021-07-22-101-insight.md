## 핵심 주장 (≤5)
1. 여러 앱을 동시에 운영할 때는 릴리즈 빌드와 스크린샷 생성 같은 반복 배포 작업을 Fastlane으로 자동화하는 편이 수작업보다 효율적이다.
2. Android Flavor와 Fastlane을 조합하면 여러 앱 변형에 대해 빌드, 스크린샷 생성, Play Console 업로드를 루프로 일괄 처리할 수 있다.
3. Fastlane 기반 스크린샷 자동화에는 가상 디바이스 설정, Espresso 테스트 코드, debug Manifest 권한 추가, Fastfile 작성이 함께 필요하다.
4. Fastlane의 `supply`와 메타데이터 디렉터리 구조를 활용하면 스크린샷뿐 아니라 앱 설명, 아이콘, 그래픽, AAB까지 Play Console에 함께 업로드할 수 있다.
5. 소수 앱에는 설정 비용이 더 클 수 있지만, 수십 개 이상 앱을 관리할 때는 자동화의 체감 효율이 크게 올라간다.

## 공개된 숫자·지표
- 제작 대상 앱 수: 85개 (기간: 작업 계획 시점, 출처: "이번에는 조금 무리해서 무려 85개의 앱을 찍어내보려고 작업하던중 겪은 삽질기를 공유하고자 합니다.")
- 앱별 빌드 시간: 평균 1분 이상 (기간: 빌드 작업 기준, 출처: "앱 개당 평균 1분 이상 소요되었기 때문에 계산적으로 모든앱을 빌드만 하는데 최소 85분 이상 시간이 소요됩니다.")
- 전체 빌드 시간: 최소 85분 이상 (기간: 85개 앱 전체 빌드 기준, 출처: "앱 개당 평균 1분 이상 소요되었기 때문에 계산적으로 모든앱을 빌드만 하는데 최소 85분 이상 시간이 소요됩니다.")
- 앱별 스크린샷 작업 시간: 최소 5분 이상 (기간: 앱 1개 기준, 출처: "앱 하나당 최소 5분 이상은 소요될 것이라 판단되었습니다.")
- 전체 스크린샷 작업 시간: 425분, 약 7시간 (기간: 85개 앱 전체 기준, 출처: "(5분 X 85개 = 425분 약 7시간)")
- 반복 작업 총소요 추정: 약 8시간 이상 (기간: 빌드+스크린샷 합산 추정, 출처: "단순하게 계산 해봤을때 위의 과정만 약 8시간 이상이 필요할것 처럼 보였습니다.")
- 테스트 라이브러리 권장 버전: `tools.fastlane:screengrab:2.1.0` (기간: 글 작성 시점, 출처: "screengrab 라이브러리는 꼭 2.1.0 버전을 사용하세요!")
- 가상 디바이스 저장공간 시도치: 20GB (기간: Flavor 다중 앱 스크린샷 작업 중, 출처: "(20GB로도 해봤는데 중간에 용량 부족을 겪었습니다 ;; 도대체 얼마나 용량을 드시는거야...)")
- Play Console 스크린샷 수 요구 언급: 기기 사이즈별 1~8장 (기간: 스토어 등록 준비 단계, 출처: "기기 사이즈별 스크린샷 1~8장")
- 애니메이션 스케일 기본값 변경: 1x -> off (기간: 가상 디바이스 안정화 설정, 출처: "(Window animatiom scale / Transition animation scale / Animator duration scale 각 기본값은 1x > off로 변경)")

## 언급된 도구·서비스
- Google Play Console: 앱 생성, API 액세스 연결, 스크린샷·메타데이터·AAB 업로드에 사용
- Admob: 앱과 매칭되는 광고 생성 및 광고 단위 키 발급에 사용
- Adsense: 계정 생성 단계에서 함께 언급됨
- Firebase: 앱 연결용 `json` 생성과 Admob 연동에 사용
- Fastlane: Android 빌드, 스크린샷 촬영, Play Store 업로드 자동화에 사용
- Homebrew: `brew install fastlane`으로 Fastlane 설치에 사용
- GitHub: 전체 코드 공개 저장소와 Fastlane 문서 PR 제출에 사용
- Google Cloud Platform: 서비스 계정 생성과 JSON 키 발급에 사용
- AVD Manager: 스크린샷 촬영용 가상 디바이스 생성에 사용
- Espresso: 화면 이동과 클릭을 포함한 스크린샷용 UI 테스트 작성에 사용
- Screengrab: 테스트 중 스크린샷 촬영과 Locale 전환에 사용
- Gradle: debug/release APK·AAB·androidTest 빌드에 사용

## 복제 가능한 전술 (≤3)
1. 반복 배포 전에 빌드/스크린샷 공수를 계산해 자동화 우선순위를 정한다.
   - 구체적 스텝: 앱 수를 확정하고, 앱당 빌드 시간과 스크린샷 촬영 시간을 직접 측정한 뒤 전체 수작업 시간을 계산한다. 그 결과가 크면 Fastlane 적용 범위를 `빌드`, `스크린샷`, `메타데이터 업로드`로 정한다.
   - 예상 리소스: 측정용 기존 프로젝트, 시간 기록, Fastlane
   - 예상 효과: 원문 기준으로 85개 앱에서는 빌드와 스크린샷만 합쳐도 약 8시간 이상 걸리는 반복 작업을 자동화 대상으로 명확히 판단할 수 있다.
2. Espresso 테스트와 Fastlane `capture_android_screenshots`를 결합해 스토어용 스크린샷을 자동 생성한다.
   - 구체적 스텝: Google APIs 이미지로 AVD를 만들고, `androidTest`에 Espresso 테스트를 작성해 캡처할 화면 흐름과 클릭 동작을 정의한다. `app/src/debug/AndroidManifest.xml`에 Screengrab 실행에 필요한 권한을 추가한 뒤, Fastfile에 `clean -> Debug assemble -> DebugAndroidTest assemble -> capture_android_screenshots` 순서의 lane을 작성하고 `fastlane screenshot`으로 실행한다.
   - 예상 리소스: Android Studio AVD, Espresso, `tools.fastlane:screengrab:2.1.0`, Fastlane, Gradle
   - 예상 효과: 원문 기준으로 앱을 일일이 설치·실행·이동하며 찍는 대신, 지정한 화면 순서대로 스크린샷을 생성하고 `fastlane/metadata/{Locale}/images/phoneScreenshots` 경로에 정리할 수 있다.
3. Flavor 리스트를 루프로 돌려 다중 앱의 빌드·스크린샷·Play Console 업로드를 일괄 처리한다.
   - 구체적 스텝: Fastfile 상단에 Flavor 배열을 정의하고, 각 lane에서 `flavors.each`로 `gradle bundle/assemble`, `capture_android_screenshots`, `supply`를 호출한다. 패키지명, APK/AAB 경로, metadata 경로를 flavor별 문자열로 조합해 분기한다.
   - 예상 리소스: Android Flavor 구성, Fastlane Fastfile, Play Console 서비스 계정 JSON 키
   - 예상 효과: 원문 기준으로 30개, 50개, 100개처럼 많은 앱을 관리할 때 빌드, 스크린샷 생성, 메타데이터 업로드를 사람 손 대신 일괄 처리할 수 있다.

## 원문 요약 (≤5문장)
글은 다수의 안드로이드 앱을 배포하면서 발생하는 빌드와 스크린샷 반복 작업을 Fastlane으로 자동화한 과정을 정리한다. 작성자는 Play Console 연동용 서비스 계정 설정부터 Espresso 테스트, debug Manifest 권한, Fastfile lane 작성까지 스크린샷 자동화의 실제 구성 요소를 단계별로 설명한다. 이어서 `supply`를 사용해 스크린샷, 설명문, 아이콘, 그래픽, AAB를 Play Console에 업로드하는 방법을 보여준다. 마지막으로 Android Flavor 배열을 루프로 처리하는 Fastfile 예시를 통해 여러 앱 변형을 한 번에 빌드·업로드하는 구조를 제안한다. 핵심 메시지는 앱 수가 많아질수록 Fastlane 설정 비용보다 자동화 효율이 커진다는 점이다.
