## 핵심 주장 (≤5)
1. Stable Diffusion 기반 서비스를 만들려면 사용자 이미지 업로드부터 LoRA 학습, 이미지 생성, 결과 저장, 알림 발송까지 전체 파이프라인을 설계해야 한다.
2. 사용자 이미지를 기반으로 BLIP 캡션을 생성한 뒤 LoRA를 학습해 개인화된 이미지 생성 서비스로 연결할 수 있다.
3. 생성 결과물과 LoRA 파일은 S3와 MySQL에 이력으로 관리하는 구조가 적합하다는 아이디어를 제시한다.
4. AWS 비용 절감을 위해 S3에 저장한 이미지와 LoRA 파일에 TTL을 적용해야 한다.
5. 전체 작업은 SQS Queue로 감싸 비동기 큐 기반으로 구현하는 것이 바람직하다고 본다.

## 공개된 숫자·지표
- 생성 이미지 수: 5 to 20 (기간: 이미지 생성 단계, 출처: "7. Create images (5 to 20)")

## 언급된 도구·서비스
- Stable Diffusion: 학습된 LoRA를 배치하고 프롬프트를 사용해 이미지를 생성하는 기반 시스템
- API: 사용자로부터 이미지를 업로드받는 입력 채널
- BLIP: 이미지로부터 캡션 txt를 생성하는 데 사용
- LoRA: 사용자 이미지 기반 학습 결과물로 생성 및 저장되는 파일
- S3: 생성 이미지와 LoRA 파일을 업로드하고 TTL을 적용하는 저장소
- MySQL: 사용자별 생성 이력을 저장하는 데이터베이스
- AWS: S3 저장 비용 절감 맥락에서 언급된 인프라
- GPU server instance: 작업 완료 후 이미지와 LoRA 파일을 삭제하는 학습/생성 서버
- SQS Queue: 전체 파이프라인을 큐로 묶어 비동기 처리하는 구성 요소

## 복제 가능한 전술 (≤3)
1. 사용자 맞춤 이미지 생성 파이프라인을 작업 큐 기준으로 분리 설계한다.
   - 구체적 스텝: 사용자 이미지 업로드 API를 만든 뒤 사용자별 디렉터리를 생성하고, BLIP로 캡션 txt를 만든 다음 LoRA 학습을 수행하고, Stable Diffusion의 Lora 폴더로 결과 파일을 이동해 프롬프트 기반 이미지를 생성한다.
   - 예상 리소스: API 서버, BLIP, LoRA 학습 환경, Stable Diffusion 실행 환경, GPU 서버
   - 예상 효과: 사용자 입력부터 개인화 이미지 생성까지 이어지는 일관된 서비스 플로우를 구현할 수 있다.
2. 결과물 저장과 이력 관리를 분리해 운영 비용과 조회 편의성을 동시에 확보한다.
   - 구체적 스텝: 생성 이미지와 LoRA 파일을 S3에 업로드하고, MySQL에는 user_id, image URL, LoRA URL을 이력으로 저장한 뒤 S3 객체에 TTL을 적용한다.
   - 예상 리소스: S3, MySQL, 업로드 로직, TTL 설정
   - 예상 효과: 사용자별 생성 이력을 관리하면서도 AWS 저장 비용 절감이 가능하다.
3. GPU 서버 정리와 완료 알림까지 포함해 비동기 잡 처리로 마감한다.
   - 구체적 스텝: 이미지 생성 후 GPU 서버 인스턴스에서 중간 이미지와 LoRA 파일을 삭제하고, 사용자에게 job done push를 보낸 뒤 전체 단계를 SQS Queue로 감싸 순차 실행되도록 구성한다.
   - 예상 리소스: GPU 서버, 푸시 알림 시스템, SQS Queue
   - 예상 효과: 리소스 정리와 사용자 알림을 자동화하고 긴 작업을 큐 기반으로 안정적으로 처리할 수 있다.

## 원문 요약 (≤5문장)
이 글은 Stable Diffusion을 활용한 서비스 아이디어를 간단한 작업 흐름 형태로 정리한 메모다. 사용자가 이미지를 업로드하면 BLIP 캡션 생성과 LoRA 학습을 거쳐 개인화 이미지 생성으로 이어지는 구조를 제안한다. 생성 결과물과 LoRA 파일은 S3에 저장하고, MySQL에는 사용자 이력을 남기는 방식을 언급한다. 저장 비용 절감을 위해 S3 객체에 TTL을 적용하고, GPU 서버의 중간 산출물은 삭제하도록 했다. 마지막으로 전체 과정을 SQS Queue로 묶어 비동기 처리하는 방향을 제시한다.
