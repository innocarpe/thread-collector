## 핵심 주장 (≤5)
1. 단순 `Thread`로 두 외부 응답을 동시에 요청하면 메인 스레드가 결과 수집을 기다리지 않고 병합을 시작해 원하지 않은 병합이 실행될 수 있다.
2. `Callable`과 `Future.get()`을 사용하면 반환값이 준비될 때까지 해당 지점에서 블락킹되어 모든 응답이 완료된 뒤 병합을 진행할 수 있다.
3. `Future`는 `Callable`이 반환한 값을 꺼내는 통로이며, 반환 전에는 `get()`이 블락킹된다는 점이 핵심 동작이다.
4. 응답 순서가 다르더라도 먼저 `get()`하는 작업이 끝나지 않으면 메인 로직은 다음 단계로 진행하지 않는다.

## 공개된 숫자·지표
- Facebook 응답 지연 예시: `0.05 * 300`초 (기간: 예제 1회 실행 기준, 출처: `/*페이스북 타임라인을 응답하는 객체 (응답시간 : 0.05 * 300) 초*/`)
- Twitter 응답 지연 예시: `0.03 * 500`초 (기간: 예제 1회 실행 기준, 출처: `/*트위터 타임라인을 응답하는 객체 (응답시간 : 0.03 * 500) 초*/`)
- Facebook 타임라인 수집 반복 수: `300`회 (기간: 예제 1회 실행 기준, 출처: `for (int i = 0; i < 300; i++) {`)
- Twitter 타임라인 수집 반복 수: `500`회 (기간: 예제 1회 실행 기준, 출처: `for ( int i = 0; i < 500; i++) {`)
- 고정 스레드풀 크기: `2` (기간: 예제 실행 시점, 출처: `ExecutorService executor = Executors.newFixedThreadPool(2);`)

## 언급된 도구·서비스
- Java `Thread`: Facebook, Twitter 요청을 각각 별도 스레드로 실행하는 비교 예시로 사용됨
- Java `Callable`: 타임라인 데이터를 반환값과 함께 비동기 처리하는 대안으로 사용됨
- Java `Future`: `Callable`의 반환값을 받고 `get()`에서 블락킹을 유도하는 객체로 설명됨
- Java `ExecutorService`: `Callable` 작업을 제출하고 실행하는 실행기 객체로 사용됨
- Java `Executors.newFixedThreadPool(2)`: Facebook과 Twitter 요청을 병렬로 실행하기 위한 고정 스레드풀 생성에 사용됨
- Facebook: 병합 대상 타임라인 데이터의 한 축으로 예시됨
- Twitter: 병합 대상 타임라인 데이터의 한 축으로 예시됨

## 복제 가능한 전술 (≤3)
1. 병합 전제 조건이 있는 다중 응답 수집은 `Thread` 대신 `Callable`과 `Future`로 감싼다.
   - 구체적 스텝: 각 외부 요청을 `Callable<List<T>>`로 구현하고 `ExecutorService`에 `submit()`한 뒤, 병합 로직에서는 각 `Future.get()`으로 결과를 받은 후에만 후속 처리를 실행한다.
   - 예상 리소스: Java 개발 환경, `java.util.concurrent` 패키지, 요청별 `Callable` 구현
   - 예상 효과: 원문처럼 두 응답이 모두 끝난 뒤에만 병합이 실행되어 불완전한 데이터 병합을 피할 수 있다.
2. 병합 시작 조건을 반환값 수집 완료와 `isDone()` 확인으로 명시한다.
   - 구체적 스텝: `futureA.get()`와 `futureB.get()`으로 각 응답을 저장하고, 두 `Future`가 모두 완료된 상태에서만 `if (futureA.isDone() && futureB.isDone())` 조건 아래 병합 루프를 실행한다.
   - 예상 리소스: 병합 클래스 1개, 응답 저장용 리스트, 완료 여부 체크 코드
   - 예상 효과: 원문 예시처럼 `Merge Start!`가 모든 응답 완료 후에만 실행되는 흐름을 만들 수 있다.
3. 메인 스레드의 진행 시점을 콘솔 로그로 검증한다.
   - 구체적 스텝: 각 `Callable` 내부에서 수집 진행 로그를 출력하고, 메인 스레드에서는 각 `get()` 직후 완료 로그와 병합 시작 로그를 남겨 블락킹 위치와 병합 시점을 확인한다.
   - 예상 리소스: 콘솔 로그 추가, 예제 실행 환경
   - 예상 효과: 원문 콘솔처럼 어느 시점에 메인 스레드가 대기하고 언제 병합이 시작되는지 눈으로 검증할 수 있다.

## 원문 요약 (≤5문장)
글은 Facebook과 Twitter 타임라인을 동시에 받아 날짜순으로 병합하는 예시로 `Thread`와 `Callable`의 차이를 설명한다. 단순 `Thread` 구현에서는 메인 스레드가 결과를 기다리지 않아 병합이 너무 일찍 시작될 수 있다고 지적한다. 대안으로 `Callable`과 `Future.get()`을 사용하면 반환값이 준비될 때까지 메인 스레드가 블락킹되어 모든 응답이 끝난 뒤 병합을 시작할 수 있다고 설명한다. 또한 `Future`는 `Callable`이 반환한 값을 받을 수 있는 객체이며, 반환 전에는 `get()`이 블락킹된다고 정리한다. 콘솔 출력 예시는 실제로 응답 수집 중에는 메인 스레드가 기다리다가 두 응답 완료 후 병합이 시작되는 흐름을 보여준다.

## 본문 포인트별 발췌
> Facebook과 Twitter의 타임라인 데이터들을 동시에 요청하여 두 타임라인데이터를 날짜별로 정렬하여 병합한다.
> 쓰레드로 구현한다면 blocking이 되지 않아 메인 쓰레드가 start()한 후 그냥 나머지 처리를 진행하게되고 결국 두 데이터가 모두 모이지 않은 상태에서 원하지 않은 병합작업이 실행되거나, 또는 처리속도가 무지막지하게 빨라서 미쳐 양쪽의 데이터가 모두 도착하지 않았는데도 병합이 완료되었다고 하는 경우가 생길 수 있다.
> Callable을 통하여 처리하게되면 이러한 점을 극복할 수 있다고 한다.
> Future클래스의 get()메서드는 해당 인스턴스가 바라보고 있는 Callable인스턴스가 처리를 마치고 값을 반환하기 전까지 Future.get()을 실행한 라인에서 블락킹 된다.
> Twitter의 응답이 종료되고 메인 쓰레드의 블락킹이 풀리게 되어 MergeStart를 콘솔에 출력하고 병합을 시작한다.
