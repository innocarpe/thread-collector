## 핵심 주장 (≤5)
1. grep으로 로그를 눈대중으로 확인하는 대신 파이프라인 명령어로 API 호출 통계를 더 확실하게 낼 수 있다.
2. `cat access.log | cut -d " " -f 7 | sort | uniq -c | sort` 명령으로 nginx 로그에서 호출된 API 경로를 집계할 수 있다.
3. 로그의 7번째 필드를 잘라내면 API 호출 경로만 따로 모아볼 수 있다.
4. 호출 횟수를 집계해 정렬하면 어떤 API가 가장 많이 호출되는지 파악할 수 있다.

## 언급된 도구·서비스
- grep: 로그파일을 대충 뒤져서 호출 수를 눈으로 파악하던 기존 방식으로 언급됨
- nginx: API 콜 기록이 남는 로그의 예시로 언급됨
- cat: 로그파일 원본 내용을 그대로 출력하는 데 사용됨
- cut: 공백 구분 기준으로 로그의 7번째 필드만 잘라내는 데 사용됨
- sort: 잘라낸 내용과 집계 결과를 정렬하는 데 사용됨
- uniq: 중복 제거와 등장 횟수 카운팅에 사용됨
- zcat: `.gz`, `.tgz` 압축 로그를 출력하는 유사 명령어로 언급됨
- xzcat: `.xz` 파일을 출력하는 유사 명령어로 언급됨
- unzip: `zip` 파일 관련 명령어로 언급됨

## 복제 가능한 전술 (≤3)
1. nginx access 로그에서 API별 호출 빈도를 셸 파이프라인으로 바로 집계한다.
   - 구체적 스텝: 로그 파일 경로를 확인한다 → `cat access.log | cut -d " " -f 7 | sort | uniq -c | sort`를 실행한다 → 출력된 경로별 호출 횟수를 보고 많이 호출된 API를 식별한다.
   - 예상 리소스: 로그 파일 1개, 리눅스 셸, `cat` `cut` `sort` `uniq`
   - 예상 효과: 어떤 API가 많이 호출되는지 로그 기반으로 확인할 수 있다.
2. 로그 구조를 먼저 확인한 뒤 필요한 필드만 잘라서 집계한다.
   - 구체적 스텝: `cat {파일명}`으로 원본 로그 한두 줄을 확인한다 → 공백 기준으로 필드가 나뉘는지 본다 → `cut -d " " -f 7`로 API 경로만 추출한다 → 이후 정렬과 카운팅을 이어 붙인다.
   - 예상 리소스: 로그 파일, 리눅스 셸, `cat`, `cut`
   - 예상 효과: 전체 로그를 눈으로 훑지 않고 분석 대상인 API 경로만 분리해 볼 수 있다.
3. 압축된 로그도 동일한 흐름으로 분석한다.
   - 구체적 스텝: 로그 압축 포맷을 확인한다 → `.gz`나 `.tgz`면 `zcat`, `.xz`면 `xzcat`으로 내용을 출력한다 → 이후 `cut`, `sort`, `uniq -c`, `sort` 순으로 이어서 집계한다.
   - 예상 리소스: 압축 로그 파일, 리눅스 셸, `zcat` 또는 `xzcat`, `cut`, `sort`, `uniq`
   - 예상 효과: 압축 로그에서도 동일한 방식으로 API 호출 통계를 낼 수 있다.

## 원문 요약 (≤5문장)
글은 서비스 운영 중 어떤 API가 많이 사용되는지 로그로 확인하는 방법을 설명한다.
작성자는 grep으로 눈대중 파악하던 방식 대신 더 확실한 집계 방법을 소개한다.
핵심은 nginx access 로그에서 공백 기준 7번째 필드를 잘라 API 경로만 추출한 뒤, 정렬과 중복 카운팅으로 호출 횟수를 집계하는 것이다.
또한 `cat`, `cut`, `sort`, `uniq -c`가 각각 어떤 역할을 하는지 간단히 풀어서 설명한다.
압축 로그를 다룰 때 쓸 수 있는 `zcat`, `xzcat`, `unzip`도 함께 언급한다.

## 본문 포인트별 발췌
> 서비스를 운영하다 어떠한 API가 많이 사용될까 궁금한적이 있었다.
> 조금더 확실한 방법을 알게되었다.
> **cat [access.log(로그파일](access.log(로그파일) PATH) | cut -d " " -f 7 | sort | uniq -c | sort**
> 호출된 수가 많은 순서대로 정렬되어서 해당 내용이 출력된다.
> 따라서 어떤 API가 가장 많이 호출되고 있는지 알 수 있다.
