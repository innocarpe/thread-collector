## 핵심 주장 (≤5)
1. RecyclerView에 이미지가 많고 이를 ViewPager 내부 여러 뷰에서 동시에 들고 있으면 메모리 사용량이 급격히 증가해 OOM 위험이 커진다.
2. Glide 로드 시 리스너로 실제 비트맵 크기와 바이트 수를 기록하면 어떤 이미지가 메모리를 많이 차지하는지 확인할 수 있다.
3. Glide의 `override(size, size)`로 로드 해상도를 줄이면 메모리 사용량을 크게 낮출 수 있다.
4. 이미지 표시 크기를 유지하려면 `ImageView`에 `adjustViewBounds="true"`를 적용해 종횡비를 유지하도록 해야 한다.
5. ViewPager 안에 무거운 요소가 많다면 UI 구조를 바꾸거나 ViewPager가 미리 로드할 범위를 조정하는 것도 대응책이다.

## 공개된 숫자·지표
- ViewPager 내 뷰 수: 4개 (기간: 해당 화면 구성 시점, 출처: "ViewPager안에 뷰가 4개가 있는데 전부 로드해서 들고 있다.")
- 메모리 사용량: 530MB가량 (기간: 문제 상황 측정 시점, 출처: "저러니 메모리가 530MB가량 차지하는것을 볼 수 있다..")
- 원본 이미지 크기: 1440 x 1440 (기간: override 적용 전, 출처: "기존에 1440 x 1440이미지를 극단적으로 200 x 200으로 줄여보았다.")
- 축소 이미지 크기: 200 x 200 (기간: override 적용 후, 출처: "기존에 1440 x 1440이미지를 극단적으로 200 x 200으로 줄여보았다.")
- 메모리 변화: 529MB > 259MB, 약 270MB 감소 (기간: override 적용 전후 비교, 출처: "529MB > 259MB 약 270MB가 감소되었다.")

## 언급된 도구·서비스
- Glide: RecyclerView 안의 이미지를 로드하고, 리스너 및 `override()` 옵션으로 메모리 사용을 확인·조절하는 데 사용
- GlideApp: `GlideApp.with(...).load(...).listener(...).override(...).into(...)` 형태로 실제 호출에 사용된 Glide API
- RecyclerView: 사진이 포함된 리스트 UI로, 이미지가 많을 때 메모리 문제가 발생한 맥락에서 언급
- ViewPager: 4개의 뷰를 모두 로드해 들고 있어 메모리 사용량 증가 원인으로 언급
- ImageView: `scaleType`, `adjustViewBounds` 설정으로 표시 크기와 비율을 다룬 UI 컴포넌트
- Android Developers 문서: `adjustViewBounds` 속성 설명을 확인한 참고 자료로 언급

## 복제 가능한 전술 (≤3)
1. Glide 로드 지점에 `RequestListener`를 붙여 실제 로드되는 비트맵의 바이트 수와 해상도를 로그로 수집한다.
   - 구체적 스텝: `onBindViewHolder`의 Glide 호출에 `.listener(...)`를 추가하고, `onResourceReady`에서 `BitmapDrawable`인지 확인한 뒤 `bitmap.byteCount`, `bitmap.width`, `bitmap.height`를 로그로 남긴다.
   - 예상 리소스: Android 코드 수정 10~20분, Glide/Logcat
   - 예상 효과: 어떤 이미지가 "말도 안되는 큰용량"으로 로드되는지 확인해 최적화 대상을 찾을 수 있다.
2. 메모리를 우선 낮춰야 할 때 Glide의 `override(width, height)`로 로드 해상도를 줄인다.
   - 구체적 스텝: 기존 Glide 체인에 `.override(200, 200)`처럼 목표 크기를 지정하고, 적용 전후 메모리 사용량을 비교한다.
   - 예상 리소스: Android 코드 수정 5~10분, Glide
   - 예상 효과: 원문 사례에서는 메모리가 `529MB > 259MB`로 줄어 약 `270MB` 감소했다.
3. 축소 로드 후에도 화면 내 표시 폭을 유지하려면 `ImageView`에 `adjustViewBounds="true"`를 적용한다.
   - 구체적 스텝: 대상 `ImageView` XML에 `android:adjustViewBounds="true"`를 추가하고, 화면에서 비율 유지와 출력 품질을 확인한 뒤 `override()` 크기를 조정한다.
   - 예상 리소스: XML 수정 5분, Android 레이아웃 점검
   - 예상 효과: 줄어든 이미지도 기존 `layout_width` 기준으로 표시되어 화면 크기를 유지하면서 출력할 수 있다.

## 원문 요약 (≤5문장)
작성자는 ViewPager 안의 여러 RecyclerView에서 많은 이미지를 동시에 로드하면서 메모리 사용량이 약 530MB까지 치솟고 OOM이 발생할 수 있는 상황을 겪었다. Glide에 리스너를 붙여 확인해보니 큰 이미지들이 한꺼번에 로드되고 있었다. 이를 줄이기 위해 `override(200, 200)`을 적용하자 메모리가 529MB에서 259MB로 감소했다. 다만 이미지 표시 크기까지 줄어드는 문제가 있어 `ImageView`에 `adjustViewBounds="true"`를 적용해 원래 레이아웃 폭에 맞춰 출력되도록 해결했다. 추가로 ViewPager 내부에 무거운 요소가 많다면 UI를 재구성하거나 로드 범위를 조정하라고 제안한다.
