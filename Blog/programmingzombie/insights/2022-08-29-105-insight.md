## 핵심 주장 (≤5)
1. Pod은 쿠버네티스 시스템의 가장 중요한 구성 요소이며, 하나의 Pod 안에 원하는 만큼의 컨테이너를 포함시킬 수 있다.
2. 레플리케이션 컨트롤러는 Pod을 복제하고 항상 실행 상태로 유지하도록 대체 Pod을 띄우는 역할을 한다.
3. 서비스는 생성과 삭제로 IP가 계속 바뀌는 Pod들을 정적 IP와 포트로 노출해 안정적으로 연결하게 해준다.
4. 쿠버네티스의 수평 확장은 원하는 Pod 개수만 선언하면 시스템이 현재 상태를 검사해 의도한 상태로 맞추는 선언적 운영 방식으로 이뤄진다.
5. 단순히 읽는 것보다 블로그에 정리하며 학습하는 방식이 장기 기억에 더 도움이 된다고 느꼈다.

## 공개된 숫자·지표
- 컨테이너 바인딩 포트 예시: 8080, 80 (기간: 해당 없음, 출처: "컨테이너 내부에는 우리가 배포한 프로세스가 있고 지정한 포트(ex. 8080, 80에) 바인딩 되어 HTTP 요청을 기다린다.")
- 스케일 아웃 후 유지한 팟 개수: 3개 (기간: 예시 실행 시점, 출처: "kubectl scale deployments/kubia --replicas=3")

## 언급된 도구·서비스
- Kubernetes: Pod, 레플리케이션 컨트롤러, 서비스, 스케일 아웃 방식 등을 학습하고 운영 개념을 이해하는 대상으로 언급됨.
- Kuberneties IN ACTION: 쿠버네티스를 정확히 이해하고 사용하기 위해 구매한 책으로 언급됨.
- Node.js: 컨테이너 내부 프로세스 예시로 언급됨.
- Java Application: 컨테이너 내부 프로세스 예시로 언급됨.
- kubectl: deployment 생성, 조회, 스케일 조정, Pod 조회에 사용한 CLI로 제시됨.
- Docker Hub: `soulduse/kubia` 이미지를 가져오는 저장소 맥락으로 언급됨.
- YAML: 회사에서 쿠버네티스 환경설정 데이터를 정의해 두는 파일 형식으로 언급됨.

## 언급된 다른 creator·앱
- soulduse/kubia: deployment 생성 예시에서 사용한 도커 이미지 이름.

## 복제 가능한 전술 (≤3)
1. 쿠버네티스 개념 학습 내용을 블로그에 기록하며 개념 단위로 재정리한다.
   - 구체적 스텝: 책이나 자료를 읽으며 Pod, 레플리케이션 컨트롤러, 서비스처럼 주제를 나누고, 각 개념의 역할과 관계를 자신의 문장으로 블로그에 정리한다.
   - 예상 리소스: 블로그 편집 도구, 학습 자료 1권
   - 예상 효과: 단순 독서보다 장기 기억에 오래 남는 학습 효과를 기대할 수 있다.
2. 간단한 deployment를 만든 뒤 `replicas` 값을 늘려 선언적 스케일 아웃을 직접 확인한다.
   - 구체적 스텝: `kubectl create deployment kubia --image=soulduse/kubia`로 배포를 만든 뒤 `kubectl get deployments`로 확인하고, `kubectl scale deployments/kubia --replicas=3` 실행 후 `kubectl get pods -o wide`로 Pod 수 증가를 검증한다.
   - 예상 리소스: Kubernetes 클러스터, `kubectl`, 도커 이미지
   - 예상 효과: 원하는 Pod 개수만 선언하면 쿠버네티스가 현재 상태를 맞추는 방식을 실습으로 이해할 수 있다.
3. 서비스 계층을 전제로 Pod 접근 구조를 설계해 동적 IP 변경 문제를 피한다.
   - 구체적 스텝: 클라이언트가 Pod에 직접 연결하지 않도록 서비스 IP와 포트를 진입점으로 두고, 요청이 서비스에 속한 Pod 중 하나로 전달되는 구조로 애플리케이션을 구성한다.
   - 예상 리소스: Kubernetes 서비스 설정, Pod 기반 애플리케이션
   - 예상 효과: Pod 생성·삭제 시 IP가 바뀌어도 서비스가 존속하는 동안 동일한 IP로 안정적으로 연결할 수 있다.

## 원문 요약 (≤5문장)
글쓴이는 쿠버네티스를 사용법만 익힌 상태에서 벗어나 정확히 이해하기 위해 책을 읽고 블로그에 정리하고 있다. 글에서는 Pod이 핵심 구성 요소이며 여러 컨테이너를 담을 수 있고, 레플리케이션 컨트롤러가 Pod을 복제해 항상 실행 상태를 유지한다고 설명한다. 또한 서비스는 자주 바뀌는 Pod IP를 정적 IP와 포트로 노출해 클라이언트가 안정적으로 접근하게 만드는 역할이라고 정리한다. 스케일 아웃 예시로 `kubectl` 명령어를 사용해 deployment를 만들고 replicas를 3개로 늘리는 과정을 보여준다. 마지막으로 정리하며 쓰는 학습 방식이 느리지만 장기 기억에는 더 도움이 된다고 회고한다.

## 본문 포인트별 발췌
> 시스템에 가장 중요한 구성 요소는 Pod이라고 함.
> Pod을 복제하고 항상 실행상태로 만드는 역할을 한다.
> 서비스가 생성되면 정적 IP를 할당받고 서비스가 존속하는 동안 IP는 변경되지 않는다.
> 유지 해야되는 팟의 개수만 정의 하였고 시스템의 의도하는 상태를 선언적으로 지정해줌으로써 쿠버네티스가 알아서 현재 상태를 검사하여 의도한 상태로 조정한다.
> 하지만 정리하며 작성하니 머릿속 장기 기억적으로는 오래 남을듯 하다.
