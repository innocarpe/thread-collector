## 핵심 주장 (≤5)
1. Kotlin의 `inner class`는 `outer class`에 대한 참조를 가지므로 외부 클래스의 멤버, 심지어 `private` 멤버에도 직접 접근할 수 있다.
2. `inner class`는 외부 클래스 인스턴스에 종속적이어서 인스턴스화할 때 외부 클래스 인스턴스가 반드시 필요하다.
3. 독립적인 동작을 수행하는 클래스가 필요할 때는 `inner class` 대신 일반 `class`를 사용하는 것이 적합하다.
4. `inner class`는 항상 외부 클래스 참조를 유지하므로 메모리 관리 측면에서 일반 클래스보다 더 많은 자원을 사용할 수 있다.
5. 상황에 맞게 `inner class`와 일반 `class`를 구분해 선택하면 더 효과적인 코드를 작성할 수 있다.

## 복제 가능한 전술 (≤3)
1. 외부 클래스 상태에 직접 접근해야 하는 경우에만 `inner class`를 선택한다.
   - 구체적 스텝: 중첩 클래스를 설계할 때 외부 클래스 멤버 접근이 필요한지 먼저 확인한다 → `private` 멤버 포함 외부 상태 접근이 필요하면 `inner class`로 선언한다 → 외부 인스턴스를 통해 `outer.Inner()` 형태로 생성해 동작을 검증한다.
   - 예상 리소스: Kotlin 코드 편집 시간, 간단한 실행 예제
   - 예상 효과: 외부 클래스 멤버에 직접 접근해야 하는 동작을 자연스럽게 구현할 수 있다.
2. 외부 클래스와 독립적으로 동작해야 하는 중첩 구조는 일반 `nested class`로 분리한다.
   - 구체적 스텝: 중첩 클래스가 외부 클래스 참조 없이도 동작 가능한지 검토한다 → 가능하면 `inner` 키워드를 제거하고 일반 `class`로 선언한다 → 기본 생성자로 생성 가능한지와 외부 멤버 접근이 막히는지를 컴파일 수준에서 확인한다.
   - 예상 리소스: Kotlin 컴파일/실행 환경, 리팩터링 시간
   - 예상 효과: 외부 클래스와의 강한 결합을 줄이고 메모리 사용 측면에서 더 가벼운 구조를 선택할 수 있다.

## 원문 요약 (≤5문장)
이 글은 Spring 사용 중 내부 클래스로 정의한 특정 클래스에서 예상치 못한 에러를 겪은 경험을 계기로 Kotlin의 `inner class`와 일반 `class` 차이를 정리한 글이다. 작성자는 `inner class`가 외부 클래스 참조를 가지므로 외부 멤버에 직접 접근할 수 있다고 설명한다. 반면 이 특성은 외부 클래스와의 강한 결합과 추가 자원 사용으로 이어질 수 있다고 본다. 또한 `inner class`는 외부 클래스 인스턴스가 있어야만 생성 가능하지만, 일반 `nested class`는 기본 생성자로 생성할 수 있다고 비교한다. 결론적으로 클래스가 독립적으로 동작해야 하면 일반 `class`를 선택하는 것이 좋다고 정리한다.

## 본문 포인트별 발췌
> inner class는 outer class에 대한 참조를 가지며, 이를 통해 outer class의 멤버에 직접 접근이 가능합니다.
> 이처럼 inner class는 outer class의 private 멤버까지도 접근이 가능합니다.
> 반대로, outer class에 대한 참조가 필요하지 않고, 독립적인 동작을 수행하는 class가 필요한 경우에는 inner class 대신 일반 class를 사용하는 것이 좋습니다.
> inner class는 외부 클래스의 인스턴스에 종속적이므로, 기본 생성자만으로는 인스턴스를 생성할 수 없습니다.
> 따라서 각 클래스가 독립적으로 동작해야 하는 경우에는 일반 class를 사용하는 것이 좋다.
