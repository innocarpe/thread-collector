## 핵심 주장 (≤5)
1. 서버 운영 중 디스크 용량 부족은 배포 실패와 서비스 마비로 이어질 수 있다.
2. `df -h`로 디스크 사용량을 확인해 원인이 저장공간 부족인지 빠르게 판단할 수 있다.
3. AWS EC2에서는 연결된 EBS 볼륨 크기를 수정해 루트 디스크 용량을 확장할 수 있다.
4. 글쓴이는 EBS 볼륨을 5GB 늘린 뒤 로그 적재, 사진 업로드, 서비스 동작이 정상화됐다고 밝혔다.

## 공개된 숫자·지표
- 디스크 사용량: Used 30G, 100% 사용중 (기간: 문제 발생 당시, 출처: "Used가 무려 30G로 100% 사용중")
- 추가 증설 용량: 5GB (기간: 볼륨 수정 시점, 출처: "5GB 더 추가 해보았다.")
- 확보된 용량: 5G (기간: 볼륨 수정 이후, 출처: "이후 용량이 5G가 확보되었고 현재는 문제 없이 로그도 잘 쌓이고, 사진업로드도 되고, 운영하던 서비스도 잘 동작하는 것을 확인 할 수 있었습니다.")

## 언급된 도구·서비스
- AWS: EC2와 EBS 볼륨 관리 화면에 접속해 디스크 용량을 늘리는 데 사용
- EC2: 용량을 늘릴 대상 서버 인스턴스로 언급
- EBS: 루트 디바이스에 연결된 볼륨을 수정해 저장공간을 확장하는 데 사용
- `df -h`: 현재 서버 디스크 사용량을 확인하는 명령어로 사용

## 언급된 다른 creator·앱
- 없음

## 복제 가능한 전술 (≤3)
1. 디스크 부족으로 서버 장애가 발생했을 때 EC2 루트 볼륨을 즉시 확장해 서비스 기능을 복구한다.
   - 구체적 스텝: 서버에서 `df -h`로 사용량을 확인한다 -> AWS 콘솔에서 `EC2`로 이동한다 -> 대상 인스턴스를 선택하고 루트 디바이스 `/dev/xvda`를 클릭한다 -> 연결된 `EBS ID`를 연다 -> 해당 볼륨의 `볼륨수정`에서 원하는 크기만큼 늘린다 -> 증설 후 서비스의 로그 적재, 업로드, 동작 상태를 확인한다
   - 예상 리소스: AWS 콘솔 접근 권한, EC2/EBS, 서버 접속 권한
   - 예상 효과: 부족한 디스크 공간을 확보해 로그 미적재, 사진 업로드 실패, 서비스 비정상 동작 문제를 완화하거나 해소할 수 있다

## 원문 요약 (≤5문장)
글쓴이는 운영 중이던 앱 서버가 마비되어 원인을 확인하던 중 저장공간 부족 문제를 발견했다. `df -h` 결과 디스크가 30G, 100% 사용 중이어서 로그 적재와 사진 업로드를 포함한 서비스 기능이 멈춘 상태였다. 불필요한 파일을 지워도 충분한 공간이 나오지 않아 AWS EC2의 루트 디바이스에 연결된 EBS 볼륨 크기를 수정했다. 글에서는 EC2 인스턴스에서 `/dev/xvda`와 EBS ID를 따라가 볼륨 수정 메뉴로 진입하는 절차를 안내한다. 작성자는 5GB를 추가한 뒤 로그, 업로드, 서비스 동작이 정상화됐다고 정리했다.

## 본문 포인트별 발췌
> 서버에 무엇인가 문제가 있어서 배포를 하려고 보니 **장치에 남은 공간이 없다**는 에러메세지가 뜨고 있는 상황이었습니다.
> 재배포도 안되고 난감한 상황이라 용량을 확인해보니 **(명령어 : df -h)**
> Used가 무려 30G로 100% 사용중
> **현재 상황에서는 로그도 쌓이지 않고, 사진업로드는 당연히 안되며 아무것도 동작하지 않는 악몽과도 같은 상황**이었습니다.
> 이후 용량이 5G가 확보되었고 현재는 문제 없이 로그도 잘 쌓이고, 사진업로드도 되고, 운영하던 서비스도 잘 동작하는 것을 확인 할 수 있었습니다.
