# programmingzombie 인사이트 핵심 글 Top 10

인사이트 밀도(핵심 주장 수 + 공개 수치 + 복제 가능 전술)를 기준으로 Top 10 선정.

## 커리어·팀 운영

## 1. 구글 캠퍼스 리쿠르팅 데이 후기 (2017.05.25)
- **파일**: 2017-05-26-54-insight.md
- **URL**: https://soulduse.tistory.com/m/54
- **핵심 주장**: 소프트웨어 엔지니어의 본질은 특정 기술 습득이 아니라 문제를 소프트웨어적으로 해결하는 능력에 있다. 좋은 해결은 코드 최적화에 그치지 않고 비즈니스와 운영 프로세스까지 다시 설계하는 접근이어야 한다. 채용과 커리어에서는 약점 보완보다 강점을 명확히 설명하는 절대평가 관점이 중요하다.
- **주목 이유**: 기술 팁을 넘어 문제 해결, 채용, 학습 방식, 자기 브랜딩까지 묶어 주는 글이다. 숫자 사례와 실행 원칙이 함께 있어 creator의 사고방식을 가장 선명하게 보여준다.

## 2. [Trello] 좋은 사이트 참고
- **파일**: 2016-03-23-26-insight.md
- **URL**: https://soulduse.tistory.com/m/26
- **핵심 주장**: 협업 툴은 TODO 기록용이 아니라 문제 발견부터 완료 후 추적까지 상태별로 관리할 수 있어야 한다. 스크럼 보드는 문제-리서치-개발-테스트-완료의 5단계로 운영하고, 이슈는 발견 즉시 카드로 기록해야 한다. 중복 문제는 Vote로 우선순위를 모아 주간 계획과 연결하는 방식이 효과적이다.
- **주목 이유**: 단순 도구 소개가 아니라 팀 운영 규칙까지 포함한 실전 협업 모델을 제시한다. 릴리즈 주기, 주간 운영, 우선순위 정리 방식이 함께 드러나 복제 가능성이 높다.

## 백엔드·인프라

## 3. Callable과 Thread
- **파일**: 2016-02-04-14-insight.md
- **URL**: https://soulduse.tistory.com/m/14
- **핵심 주장**: 단순 `Thread`만으로 병렬 요청을 처리하면 메인 스레드가 결과 수집을 기다리지 않아 병합이 너무 일찍 실행될 수 있다. `Callable`과 `Future.get()`을 쓰면 반환값이 준비될 때까지 필요한 지점에서 블로킹해 전체 응답 완료 후 후속 로직을 실행할 수 있다. 병합 전제 조건이 있는 멀티소스 처리에는 이 구조가 더 안전하다.
- **주목 이유**: 동시성 개념을 추상 설명이 아니라 Facebook/Twitter 타임라인 병합 사례로 풀어낸다. 병렬 처리와 결과 수집 타이밍을 어떻게 설계해야 하는지 명확한 기준을 준다.

## 4. Timer 클래스를 이용한 작업 스케쥴링
- **파일**: 2015-12-12-5-insight.md
- **URL**: https://soulduse.tistory.com/m/5
- **핵심 주장**: `Timer`와 `TimerTask`만으로도 자바에서 특정 시각 또는 주기 작업을 간단히 구성할 수 있다. `schedule()`은 fixed-delay, `scheduleAtFixedRate()`는 fixed-rate이므로 실행 정확도 요구에 따라 구분해서 써야 한다. 타이머 스레드가 앱 종료를 막지 않게 하려면 `cancel()` 또는 `Timer(true)` 같은 종료 전략도 함께 고려해야 한다.
- **주목 이유**: API 소개를 넘어 실행 정확도와 프로세스 종료 조건까지 짚는다. 배치성 작업과 주기 실행을 설계할 때 바로 적용할 수 있는 판단 기준이 있다.

## 5. [Java-lib] Object Mapper에 대해서
- **파일**: 2016-03-08-22-insight.md
- **URL**: https://soulduse.tistory.com/m/22
- **핵심 주장**: JSON 처리는 수동 파싱이나 GSON보다 `ObjectMapper` 중심으로 정리하면 더 깔끔하고 단순해질 수 있다. 핵심은 `writeValueAsString()`, `readValue()`, `TypeReference`를 이용해 객체, `Map`, `List<Map>`를 일관되게 변환하는 것이다. Jackson 매핑 클래스의 기본 생성자 유지 같은 실무 주의점도 중요하다.
- **주목 이유**: 단순 라이브러리 메모가 아니라 JSON 처리 코드를 어떻게 표준화할지에 대한 방향을 준다. API 요약, 버전 정보, 컬렉션 역직렬화 팁까지 밀도가 높다.

## 6. 리눅스 서버 셋팅
- **파일**: 2016-02-15-17-insight.md
- **URL**: https://soulduse.tistory.com/m/17
- **핵심 주장**: 우분투 가상머신에 `openssh-server`를 설치하면 Windows 환경에서 SSH로 원격 접속하는 기본 서버 운영 기반을 만들 수 있다. 이후 `/etc/network/interfaces`로 정적 IP를 고정하고 `/etc/ssh/sshd_config`로 포트, 인증, PAM, DNS 같은 접속 정책을 직접 관리할 수 있다. 서버 설정은 한 번 끝내는 작업이 아니라 재참조 가능한 운영 메모로 남겨야 한다.
- **주목 이유**: 이 글은 실제 설정값과 파일 경로가 그대로 담겨 있어 인프라 초반 셋업 문서로 바로 쓸 수 있다. 서버 접속, 네트워크 고정, SSH 정책 조정까지 한 흐름으로 연결된다.

## 7. [MySQL] 우분투 설정 및 HeidiSQL연동
- **파일**: 2016-04-11-28-insight.md
- **URL**: https://soulduse.tistory.com/m/28
- **핵심 주장**: MySQL은 로컬 윈도우 설치 대신 우분투 가상머신에 두고 외부 접속이 가능하도록 구성해 관리 도구와 연결할 수 있다. 원격 접속을 열려면 사용자 권한 부여와 `my.cnf`의 `bind-address` 수정이 함께 필요하다. DB와 테이블 생성까지 포함해 개발용 데이터 저장소를 빠르게 마련하는 것이 핵심이다.
- **주목 이유**: DB 설치, 원격 연결, 스키마 생성이 한 문서에 묶여 있어 실무 셋업 가치가 높다. Spring 연동 전 단계의 개발 환경 준비 과정을 구체적으로 재현할 수 있다.

## 8. [안드로이드] Android Studio에서 Junit 기반 단위 테스트(Unit Test) 환경 구축하기
- **파일**: 2016-07-27-46-insight.md
- **URL**: https://soulduse.tistory.com/m/46
- **핵심 주장**: 안드로이드 Instrumentation 테스트는 느리기 때문에, 빠른 피드백이 필요하면 로컬 Junit 테스트 환경을 별도로 갖추는 편이 낫다. `src/test/java` 구조, Gradle 의존성 추가, `Build Variant`의 `Test Artifact` 설정이 핵심 구성 요소다. 간단한 계산 예제로 먼저 환경을 검증하는 접근이 유효하다.
- **주목 이유**: 안드로이드 테스트 환경을 실제 IDE 기준 절차로 정리해 둔 드문 실전형 글이다. 로컬 테스트 전환의 이유와 설정 순서가 명확해서 재사용성이 높다.

## 9. [안드로이드] Fragment BackStack에 대한 고찰.
- **파일**: 2016-03-10-23-insight.md
- **URL**: https://soulduse.tistory.com/m/23
- **핵심 주장**: `AppCompatActivity` 기반 화면에서 `addToBackStack()`이 기대대로 동작하지 않으면 `getFragmentManager()` 대신 `getSupportFragmentManager()`를 써야 한다. 이때 Fragment import와 상속 타입도 support 계열로 맞춰야 한다. 최소 지원 버전에 따라 정식 Fragment와 support v4 전략을 분리하는 판단이 필요하다.
- **주목 이유**: 흔한 시행착오를 원인-해결-호환성 판단까지 연결해 정리한 글이다. 안드로이드 화면 전환 구조를 다루는 사람에게 바로 도움이 되는 문제 해결형 인사이트다.

## 10. [Spring] @Transactional 에 관해
- **파일**: 2016-05-18-40-insight.md
- **URL**: https://soulduse.tistory.com/m/40
- **핵심 주장**: 선언적 트랜잭션을 쓰려면 `transactionManager` 등록과 `<tx:annotation-driven>` 설정이 먼저 갖춰져야 한다. 이후 `isolation`, `propagation`, `readOnly`, `rollbackFor`, `timeout` 등 속성으로 트랜잭션의 간섭 허용 범위와 실행 방식을 세밀하게 조절할 수 있다. 트랜잭션은 여러 처리 단계를 하나의 성공 또는 전체 롤백으로 묶는 안전장치다.
- **주목 이유**: 설정법과 개념 설명이 함께 있어 입문용 참고문서로 밀도가 높다. 서비스 계층 설계에서 자주 놓치는 격리 수준과 전파 옵션을 한 번에 정리해 준다.
