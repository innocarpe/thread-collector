## 핵심 주장 (≤5)
1. API Gateway로 `multipart/form-data`를 그대로 Lambda에 넘기는 구성은 추가 설정 없이는 바로 동작하지 않는다.
2. Lambda 배포 패키지에 `Pillow`를 직접 포함하면 `cannot import name '_imaging' from 'PIL'` 오류가 발생할 수 있어, 작성자는 레이어로 분리해 해결했다.
3. API Gateway에서 `multipart/form-data`를 받으려면 이진 미디어 형식과 통합 요청의 매핑 템플릿 설정이 필요하다.
4. API Gateway를 거친 `multipart/form-data`는 Lambda에서 `body-json` 기준으로 base64 디코딩 후 다시 파싱하는 처리가 필요하다.
5. 파싱한 이미지 데이터를 S3에 업로드한 뒤 Step Functions 실행으로 후속 처리를 연결할 수 있다.

## 공개된 숫자·지표
- Python 버전: 3.9 (기간: 해당 구현 시점, 출처: "저는 python 3.9 버전을 기준으로 사용하였고 각자 파이썬 버전에 맞는 버전을 찾아서 ARN을 추가 해주시면 됩니다.")

## 언급된 도구·서비스
- AWS API Gateway: `multipart/form-data` 요청을 받아 Lambda로 전달하는 API 엔드포인트로 사용
- AWS Lambda: 업로드 요청을 처리하고 멀티파트 데이터 파싱, 이미지 변환, S3 업로드, Step Functions 실행을 담당
- Amazon S3: 업로드된 이미지를 저장하는 버킷으로 사용
- AWS Step Functions: 이미지 업로드 후 후속 워크플로를 실행하는 데 사용
- Pillow (PIL): 이미지 포맷 확인과 JPEG 변환에 사용
- boto3: S3와 Step Functions 클라이언트를 생성하는 AWS SDK로 사용
- shortuuid: S3 저장 경로용 고유 폴더명을 만드는 데 사용
- Postman: 생성한 API Gateway 호스트로 파일 전송 테스트에 사용
- Klayers: Pillow를 Lambda 레이어로 추가하기 위한 ARN 참조 소스로 사용

## 언급된 다른 creator·앱
- Postman: "Postman으로 생성한 API Gateway host로 파일전송" 문맥에서 테스트 도구로 언급

## 복제 가능한 전술 (≤3)
1. Lambda 배포 패키지에서 `Pillow`를 제외하고 이미지 라이브러리는 Lambda Layer로 분리한다.
   - 구체적 스텝: `pip install --target ./package boto3 shortuuid`로 필요한 라이브러리만 설치해 zip으로 배포하고, Lambda 콘솔의 `add layer`에서 Python 버전에 맞는 `Klayers`의 Pillow ARN을 추가한다.
   - 예상 리소스: AWS Lambda 콘솔 접근, 패키징 시간 약간, `boto3` `shortuuid` `Pillow layer`
   - 예상 효과: `cannot import name '_imaging' from 'PIL'` 오류를 피하면서 Lambda에서 Pillow를 사용할 수 있다.
2. API Gateway가 `multipart/form-data`를 받도록 바이너리 미디어와 통합 요청을 함께 설정한다.
   - 구체적 스텝: API Gateway 콘솔에서 API 선택 후 설정으로 들어가 이진 미디어 형식에 `multipart/form-data`를 추가하고, 리소스 탭의 해당 메서드에서 통합 요청을 열어 Lambda 함수 연결과 매핑 템플릿 저장을 수행한다.
   - 예상 리소스: AWS API Gateway 콘솔, 설정 작업 시간 약간
   - 예상 효과: `multipart/form-data` 요청 본문이 Lambda에서 처리 가능한 `body-json` 형태로 넘어온다.
3. Lambda에서 멀티파트 요청을 `base64` 디코딩 후 파싱하고, JPEG가 아닌 이미지는 변환해 S3에 저장한다.
   - 구체적 스텝: 이벤트의 `body-json`을 `base64.b64decode`로 디코딩하고, `Content-Type` 헤더에서 boundary를 꺼내 `cgi.parse_multipart`로 `title`과 `images_data`를 분리한 뒤, `is_jpeg` 검사 결과에 따라 Pillow로 JPEG 변환 후 `s3_client.put_object`로 업로드하고 필요하면 `step_functions_client.start_execution`을 호출한다.
   - 예상 리소스: Lambda 코드, `cgi` `base64` `io` `Pillow` `boto3`, S3 버킷, Step Functions 상태 머신
   - 예상 효과: 여러 장의 이미지를 API Gateway 경유로 받아 S3에 저장하고 후속 워크플로까지 연결할 수 있다.

## 원문 요약 (≤5문장)
작성자는 API Gateway로 여러 장의 이미지를 받아 Lambda를 통해 S3에 업로드하는 과정에서 두 가지 핵심 문제를 겪었다. 첫째는 Lambda 배포 시 `Pillow`를 직접 포함하면 import 충돌이 발생하는 문제였고, 이를 배포 패키지에서 제외한 뒤 Lambda Layer로 추가해 해결했다. 둘째는 API Gateway가 `multipart/form-data`를 그대로 받지 못하는 문제였고, 이진 미디어 형식과 매핑 템플릿 설정으로 해결했다. 이후 Lambda에서 base64 디코딩과 멀티파트 파싱을 거쳐 이미지를 S3에 저장하고 Step Functions를 실행하는 전체 코드를 제시했다. 마지막으로 Postman으로 파일 전송 테스트를 수행해 S3 저장 결과를 확인했다.

## 본문 포인트별 발췌
> API Gateway로 이미지를 여러장 전송해서 Lambda를 통해 S3에 이미지를 업로드할 일이 생겼다.
> AWS 공식 홈페이지에서 어떻게 Lambda에 라이브러리와 함께 업로드 하면되는지 링크를 참조해서 해결하였는데 PIL(Pillow) 라이브러리를 사용하는 경우 AWS Lambda: cannot import name '_imaging' from 'PIL' 라는 에러가 발생한다.
> AWS API Gateway가 multipart/form-data 형식으로 데이터를 던지면 받아주질 못합니다.
> 이제 API로 데이터를 전송하면 body-json으로 데이터가 넘어오게 됩니다.
> 그런데 이게 좀 복잡한게 넘어온 데이터가 base64러 인코딩 되어 이걸 다시 base64로 디코딩해주고 파싱을 하는 과정을 거쳐야 합니다.
