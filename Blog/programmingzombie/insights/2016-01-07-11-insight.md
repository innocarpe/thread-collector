## 핵심 주장 (≤5)
1. `InterruptedException`를 빈 `catch`로 무시하면 스레드의 interrupted 상태 정보가 사라질 수 있으므로 권장되지 않는다.
2. `InterruptedException`를 잡았을 때는 `Thread.currentThread().interrupt();`로 interrupted 상태를 다시 설정하고 현재 작업을 종료하는 처리가 올바른 동작이다.
3. 해당 예외가 발생하면 안 되는 문맥이라면 `catch` 블록에서 `assert false;`로 재사용 시점의 잘못된 환경을 드러내는 방법이 더 나을 수 있다.

## 언급된 도구·서비스
- Java `Thread.sleep`: 스레드를 일시 정지시키는 메서드로, `InterruptedException` 처리 맥락에서 설명됨
- Java `Thread.currentThread().interrupt()`: `InterruptedException` 발생 후 interrupted 상태를 다시 설정하는 코드로 제시됨
- `assert`: 예외가 절대 발생하면 안 되는 경우를 검증하는 대안으로 제시됨
- Stack Overflow: 글의 출처로 링크됨

## 복제 가능한 전술 (≤3)
1. `InterruptedException`를 무시하지 말고 인터럽트 상태를 복원한 뒤 작업 종료 경로를 명시한다.
   - 구체적 스텝: `catch (InterruptedException e)` 블록에서 `Thread.currentThread().interrupt();`를 호출하고, 바로 이어서 현재 작업을 중단하거나 메서드 실행을 종료하는 코드를 둔다.
   - 예상 리소스: Java 코드 수정, 예외 처리 검토 시간
   - 예상 효과: interrupted 상태 유실을 막고 스레드 종료 의도를 코드에 반영할 수 있다.
2. 해당 코드가 인터럽트되면 안 되는 전제라면 `assert false;`로 재사용 위험을 노출한다.
   - 구체적 스텝: `catch (InterruptedException e)` 블록에 `assert false;`를 넣어, 인터럽트가 발생하는 환경에서 즉시 이상을 드러내게 한다.
   - 예상 리소스: Java 코드 수정, 실행 환경의 assert 사용 여부 점검
   - 예상 효과: 원래는 발생하지 않아야 하는 예외가 다른 환경에서 발생할 때 조기에 감지할 수 있다.

## 원문 요약 (≤5문장)
글은 `Thread.sleep` 사용 중 발생할 수 있는 `InterruptedException`를 어떻게 처리해야 하는지 다룬다. 원문은 단일 스레드 앱과 일부 멀티스레드 앱에서는 이 예외 처리 경로가 거의 실행되지 않을 수 있다고 설명한다. 하지만 빈 `catch`로 무시하면 스레드의 interrupted 상태가 지워져 중요한 정보가 사라질 수 있다고 경고한다. 권장안으로는 `Thread.currentThread().interrupt();`로 상태를 복원하고 현재 작업을 끝내는 방식을 제시한다. 대안으로, 해당 예외가 절대 발생하면 안 되는 코드라면 `assert false;`를 두어 재사용 환경의 문제를 드러내라고 제안한다.
