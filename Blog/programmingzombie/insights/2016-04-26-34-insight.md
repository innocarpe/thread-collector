## 핵심 주장 (≤5)
1. SQLite 테이블에서 현재 시간을 기본값으로 넣을 때 `datetime('now')`나 `CURRENT_TIMESTAMP`를 쓰면 한국 시간보다 9시간 빠르게 저장될 수 있다.
2. `datetime('now','localtime')`를 사용하면 우리나라 현재 시간과 동일한 시간을 얻을 수 있다.
3. `datetime('now')`와 `CURRENT_TIMESTAMP`는 국제표준시간(UTC) 기준 값을 출력한다.

## 공개된 숫자·지표
- 시차: 9시간 (기간: 해당 없음, 출처: "우리나라 현재시간과 비교했을 때 9시간이 빠르게 나오는 현상이 발생했다.")
- 예시 현재 시간: 2016-04-26 17:46:49 (기간: 예시 시점, 출처: "EX) 현재 우리나라 시간이 2016-04-26 17:46:49라고 했을때,")

## 언급된 도구·서비스
- SQLite: 테이블 생성 시 `DATETIME` 기본값을 설정하고 현재 시간 저장 동작을 비교하는 데 사용됨
- 안드로이드: SQLite 관련 작업이 이뤄진 개발 환경 맥락으로 언급됨

## 복제 가능한 전술 (≤3)
1. SQLite에서 로컬 시간 기준으로 생성 시각을 저장해야 하면 `DATETIME DEFAULT (datetime('now','localtime'))`를 기본값으로 사용한다.
   - 구체적 스텝: 테이블 생성 SQL에서 날짜 컬럼의 기본값을 `datetime('now')` 또는 `CURRENT_TIMESTAMP` 대신 `datetime('now','localtime')`로 지정하고, 삽입 후 저장값이 로컬 시간과 일치하는지 확인한다.
   - 예상 리소스: SQLite 테이블 스키마 수정, 테스트용 INSERT/조회 몇 분
   - 예상 효과: 원문 예시처럼 UTC 기준으로 9시간 빠르게 저장되는 문제를 피하고 우리나라 현재 시간과 동일한 값을 얻을 수 있다.

## 원문 요약 (≤5문장)
글은 안드로이드에서 SQLite 테이블의 날짜 기본값을 설정할 때 발생한 9시간 시차 문제를 설명한다.
작성자는 `datetime('now','localtime')`, `datetime('now')`, `CURRENT_TIMESTAMP`의 출력 차이를 예제로 비교했다.
비교 결과 `datetime('now','localtime')`만 한국 현재 시간과 동일했고, 나머지 둘은 UTC 기준 시간으로 저장됐다.
따라서 로컬 시간 기준 저장이 필요하면 `localtime` 옵션을 써야 한다는 점을 정리한다.

## 본문 포인트별 발췌
> SQLite 관련 작업중 테이블에 현재 시간을 기본설정 값으로 넣으면
> 우리나라 현재시간과 비교했을 때 9시간이 빠르게 나오는 현상이 발생했다.
> datetime('now','localtime')    // => 현재 우리나라 시간과 동일한 시간을 얻는다.
> datetime('now')                        // => 국제표준시간(UTC) 기준의 값을 출력
> CURRENT_TIMESTAMP              // => 국제표준시간(UTC) 기준의 값을 출력
