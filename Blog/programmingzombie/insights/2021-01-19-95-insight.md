## 핵심 주장 (≤5)
1. 이번 글은 구글이 권장하는 최신 방식인 FCM HTTP v1 API를 사용해 푸시를 발송하는 방법을 정리한다.
2. Firebase 문서 기준으로 FCM Legacy에서 HTTP v1으로 마이그레이션하면 액세스 토큰 기반 보안 향상, 플랫폼별 메시지 맞춤설정, 확장성 강화 이점이 있다.
3. 작성자는 HTTP v1의 가장 큰 장점으로 iOS, Android, Web 등 플랫폼별 분기처리가 쉬워지는 점을 꼽는다.
4. OAuth 2.0 보안 모델에서는 액세스 토큰 수명이 짧아 토큰 노출 시에도 최대 1시간 이후에는 사용할 수 없다고 설명한다.
5. OAuth 2.0 Playground에서 발급한 Access token은 Postman과 Terminal에서도 재사용해 FCM HTTP v1 요청을 보낼 수 있다.

## 공개된 숫자·지표
- 액세스 토큰 사용 가능 시간: 최대 1시간 이후 사용 불가 (기간: 토큰 노출 이후, 출처: "그리고 추가적인 보안향상은 덤 👏 (OAuth2 보안 모델에 따라 세스 토큰 수명이 짧기 때문에 토큰이 노출되더라도 최대 1시간 이후 사용 불가)")

## 언급된 도구·서비스
- FCM HTTP v1 API: 최신 방식의 푸시 발송 API로 소개되며 마이그레이션 대상이다.
- Firebase Console: 프로젝트 ID를 확인하고 복사하는 용도로 사용된다.
- Firebase Cloud Messaging API v1: OAuth 2.0 Playground의 권한 승인 대상 API로 사용된다.
- OAuth 2.0 Playground: 권한 승인, 토큰 발급, API 요청 테스트에 사용된다.
- Postman: Playground에서 발급한 Access token으로 FCM HTTP v1 요청을 보내는 도구로 사용된다.
- Terminal: `curl`로 FCM HTTP v1 요청을 보내는 도구로 사용된다.
- Access token: Authorization 헤더의 Bearer 토큰으로 사용된다.
- Refresh token: OAuth 2.0 Playground의 토큰 교환 단계에서 함께 생성된다.

## 복제 가능한 전술 (≤3)
1. OAuth 2.0 Playground로 FCM HTTP v1 테스트용 액세스 토큰을 발급받아 직접 API를 호출한다.
   - 구체적 스텝: Firebase Console에서 프로젝트 ID를 복사하고, OAuth 2.0 Playground에서 Firebase Cloud Messaging API v1을 선택해 권한을 승인한 뒤, authorization code를 token으로 교환한다. 생성된 Access token을 사용해 `https://fcm.googleapis.com/v1/projects/<프로젝트 ID>/messages:send`로 POST 요청을 보낸다.
   - 예상 리소스: Google 계정, Firebase 프로젝트, OAuth 2.0 Playground, FCM 토큰
   - 예상 효과: 별도 서버 코드 없이 FCM HTTP v1 요청이 성공하는지 바로 검증할 수 있다.
2. Postman으로 FCM HTTP v1 요청 템플릿을 만들어 수동 발송 절차를 반복 가능하게 만든다.
   - 구체적 스텝: Postman에서 POST 요청을 생성하고 Request URI를 FCM HTTP v1 엔드포인트로 설정한다. `Authorization: Bearer <Access token>`, `Content-Type: application/json` 헤더를 추가하고, 본문에 `message.token`과 `notification.title`, `notification.body`를 포함한 JSON을 넣어 전송한다.
   - 예상 리소스: Postman, Access token, 프로젝트 ID, 대상 기기의 FCM_TOKEN
   - 예상 효과: 토큰과 메시지 본문만 교체하며 반복 테스트할 수 있다.
3. Terminal의 `curl` 명령으로 동일한 FCM HTTP v1 요청을 자동화 가능한 형태로 실행한다.
   - 구체적 스텝: `curl --location --request POST`로 FCM HTTP v1 엔드포인트를 호출하고, Authorization과 Content-Type 헤더를 지정한다. `--data-raw`에 `message.token`, `notification.title`, `notification.body`를 담은 JSON 본문을 넣어 실행한다.
   - 예상 리소스: Terminal, curl, Access token, 프로젝트 ID, FCM_TOKEN
   - 예상 효과: GUI 도구 없이도 동일한 요청을 재현하고 스크립트화할 수 있다.

## 원문 요약 (≤5문장)
이 글은 FCM Legacy 대신 FCM HTTP v1 API를 사용해 푸시 알림을 보내는 방법을 정리한 Part 2다. 작성자는 HTTP v1의 장점으로 액세스 토큰 기반 보안, 플랫폼별 메시지 맞춤설정, 확장성을 소개하며 특히 플랫폼 분기처리가 쉬워진다고 본다. 실습 흐름은 Firebase Console에서 프로젝트 ID를 확인하고 OAuth 2.0 Playground에서 Firebase Cloud Messaging API v1 권한을 승인해 Access token을 발급받는 것이다. 발급한 토큰은 OAuth 2.0 Playground 자체, Postman, Terminal의 `curl` 요청에서 공통으로 활용할 수 있다. 요청 본문은 `message.token`과 `notification.title`, `notification.body`를 포함한 JSON 구조를 사용한다.
