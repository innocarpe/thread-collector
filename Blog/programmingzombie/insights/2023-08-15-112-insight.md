## 핵심 주장 (≤5)
1. mustache 화면에서 CSS 파일이 `static/css` 경로에 있어도 `No mapping for GET /css/gallery.css` 오류가 발생할 수 있다.
2. 이 문제의 원인은 `WebMvcConfigurer` 구현 클래스에 추가한 `@EnableWebMvc` 설정이었다.
3. `@EnableWebMvc`를 활성화하면 스프링 부트의 기본 정적 리소스 처리와 자동 구성 리소스 핸들러가 비활성화된다.
4. `@EnableWebMvc`를 유지하려면 `addResourceHandlers`를 재정의해 `classpath:/static/` 경로를 수동으로 등록해야 한다.
5. `addResourceHandlers`를 추가한 뒤 정적 리소스가 정상적으로 동작하는 것을 확인했다.

## 공개된 숫자·지표
- corePoolSize: 10 (기간: 없음, 출처: `executor.corePoolSize = 10`)
- maxPoolSize: 50 (기간: 없음, 출처: `executor.maxPoolSize = 50`)
- queueCapacity: 100 (기간: 없음, 출처: `executor.setQueueCapacity(100)`)

## 복제 가능한 전술 (≤3)
1. Spring Boot에서 정적 리소스 404/매핑 오류가 나면 `@EnableWebMvc` 사용 여부를 먼저 점검한다.
   - 구체적 스텝: `WebMvcConfigurer` 구현 클래스와 설정 클래스를 열어 `@EnableWebMvc` 존재 여부를 확인하고, 정적 파일이 `src/main/resources/static`에 있어도 `/css/...` 요청이 실패하는지 로그로 검증한다.
   - 예상 리소스: 코드 검색 도구, Spring 설정 파일, 서버 로그
   - 예상 효과: 원문 사례처럼 정적 리소스 자동 구성 비활성화 여부를 빠르게 식별할 수 있다.
2. `@EnableWebMvc`를 유지해야 한다면 정적 리소스 핸들러를 명시적으로 등록한다.
   - 구체적 스텝: `WebMvcConfigurer`에 `addResourceHandlers`를 override하고 `registry.addResourceHandler("/**").addResourceLocations("classpath:/static/")`를 추가한 뒤 CSS 요청 재호출로 동작을 확인한다.
   - 예상 리소스: MVC 설정 클래스, 애플리케이션 재기동, 브라우저 또는 서버 로그
   - 예상 효과: 원문에서처럼 `src/main/resources/static` 하위 리소스를 정상 제공할 수 있다.

## 원문 요약 (≤5문장)
작성자는 mustache로 화면을 구성한 뒤 `/css/gallery.css` 링크가 적용되지 않고 `No mapping for GET /css/gallery.css` 오류가 발생하는 문제를 겪었다. CSS 파일은 `static/css` 경로에 정상적으로 존재했지만, 과거에 작성한 `WebMvcConfigurer` 구현 코드가 원인이었다. 특히 `@EnableWebMvc`가 스프링 부트의 기본 정적 리소스 자동 구성을 비활성화하고 있었다. 이를 해결하기 위해 `addResourceHandlers`를 재정의해 `classpath:/static/`를 수동 등록했다. 이후 리소스 로딩이 정상 동작함을 확인했다.

## 본문 포인트별 발췌
> 문제가 되는 부분이 있었는데 아래 CSS link가 안먹히는게 아닌가?
> 그럼에도 불구하고 페이지에 진입시 서버에서 다음과 같은 에러를 출력하고 있었다.
> 원인이 무엇일까 한참헤멨는데 내가 예전에 설정한 WebMvcConfigurer을 상속받아 구현한 로직에 문제가 있었다.
> `@EnableWebMvc`는 Spring MVC의 자동 구성을 사용 중지하며, 이 어노테이션을 활성화하면 스프링 부트에 의해 자동으로 구성되는 모든 리소스 핸들러를 비활성화 한다고 한다.
> 아래 부분을 추가해주었고 리소스 가져오기가 잘 동작함을 확인하였다.
