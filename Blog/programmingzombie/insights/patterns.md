# programmingzombie 반복 패턴 분석

## 반복 사용 전술
- 문제-재현-해결 메모화: 개인 앱이나 실무 개발 중 막힌 지점을 아주 짧은 해결 글로 남기는 패턴이 가장 강하다. 안드로이드 네트워크 체크, Fragment `Snackbar`, RecyclerView `CheckBox`, SQLite 시간 오차, WebView 영상 재생, SourceTree 실행 불가, Intellij 경고 해제처럼 실제 에러 상황을 바로 재현하고 해결 코드나 설정만 남긴 글이 15회 이상 등장한다.
- 외부 레퍼런스 선별 후 재정리: 직접 장문 튜토리얼을 쓰기보다 먼저 “좋은 자료”를 발견해 링크를 저장하고, 필요하면 나중에 따라 해보는 흐름이 반복된다. NodeJS 사이트, JavaScript 학습 사이트, 안드로이드 블로그, Spring Boot, Spring JDBC, socket.io 예제, Trello 운영법 등 최소 8회 이상 같은 방식이 보인다.
- 설정 체크리스트화: 구현보다 환경설정, 권한, 라이브러리, 명령어, 관리자 경로를 빠르게 찾게 만드는 정리 습관이 일관적이다. `AndroidManifest.xml` 퍼미션, MySQL 우분투 설정, adb 버전 확인, 시스템 앱 설치, SQLite 추출, Timer/ScheduledExecutorService 비교처럼 “다음에도 바로 쓰려는” 보관형 글이 많다.

## 일관되게 사용하는 도구
- Android/Java: 50개 대상 글 중 안드로이드 제목 글만 24개로 핵심 축이다. UI, 네트워크, SQLite, 퍼미션, WebView, Fragment, RecyclerView까지 대부분을 Android와 Java 조합으로 해결한다.
- Stack Overflow와 구글링: `Stack Overflow`가 7개, `구글/구글링`이 8개 파일에서 직접 언급된다. 해결책을 처음부터 발명하기보다 검증된 답을 빨리 찾고, 자기 상황에 맞게 다시 메모하는 방식이다.
- SQLite와 adb: 개인 앱 데이터를 저장·검증·추출하는 흐름에서 반복 등장한다. 저장 구조를 만들고, 기기에서 DB를 꺼내 보고, 시간대 문제까지 보정하는 식으로 로컬 데이터 운용 역량을 쌓는다.
- Trello·SourceTree·Intellij 같은 주변 도구: 코딩 자체뿐 아니라 협업 보드, Git GUI, IDE 경고 처리까지 생산성 도구를 같이 다룬다.

## 반복되는 수익화 패턴
- 직접 수익화 부재, 간접 자산 축적형: 광고 매출, 구독, 판매 같은 직접 수익화 언급은 없다. 대신 검색형 문제 해결 글을 축적해 기술 신뢰도와 포트폴리오를 쌓는 구조가 반복된다.
- 취업 경쟁력 자본화: 2017년 글에서 이 흐름이 명확해진다. 작은 해결 글들의 누적은 단기 매출보다 “나는 어떤 문제를 풀 수 있는 사람인가”를 증명하는 자산으로 기능한다. 공개 수치는 없지만, 커리어를 수익화의 핵심 통로로 보는 패턴은 분명하다.

## 핵심 믿음·철학
- 기술보다 문제 해결이 우선이라는 믿음이 후반으로 갈수록 선명해진다.
- 반복 삽질을 줄이기 위해 자신이 겪은 시행착오를 바로 기록한다.
- 실제 인용구:
> 소프트웨어 엔지니어는 주어진 문제를 소프트웨어적으로 해결하는 사람.
> 소프트웨어의 방점이 있는 것이 아니라 사실은 문제라는 것에 방점이 있다.
> 좋은 아이디어를 원하는 것이 아니라,  될 때까지 해결하는 것이 중요하다.
> 저걸 몰라서 1시간이나 삽질했다 ㅠㅠ 나 처럼 삽질 하는 사람이 없길 바라며 글을 포스팅한다.

## 성장 궤적
- 2015년 말에는 Java, Android, 템플릿 엔진, 기초 네트워크처럼 학습 초기의 넓은 탐색이 보인다.
- 2016년 초중반에는 Android 앱 개발의 실제 난관으로 무게중심이 이동한다. SQLite, Fragment, 액션바, 퍼미션, 패키지명, WebView, socket.io처럼 “내 앱을 굴리기 위한” 실전형 지식이 축적된다.
- 같은 시기 Spring, MySQL, Ubuntu, Trello를 건드리며 앱 개발자에서 서버·협업·운영 감각까지 넓힌다.
- 2017년에는 단순 팁 정리를 넘어 왜 개발자가 성장하는지에 대한 메타 관점이 생긴다. 즉, programmingzombie의 궤적은 링크 수집가에서 문제 해결 아카이버로, 다시 문제 해결 철학을 언어화하는 개발자로 진화한다.
