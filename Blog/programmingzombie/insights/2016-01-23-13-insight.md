1. JavaScript에서는 동일한 이름의 함수를 다시 선언하면 아래에 선언한 함수가 앞의 함수를 덮어쓴다.

3. 기본형 값 비교에서는 `Call by Value`가 적용되어 복사본의 값이 독립적으로 유지된다.

## 복제 가능한 전술 (≤3)
1. 함수 재선언 동작을 직접 검증하는 예제를 작성해 언어 특성을 확인한다.
   - 구체적 스텝: 동일한 이름의 함수를 두 번 선언하고 각 함수 내부 프로퍼티 값을 다르게 둔 뒤, 실제 호출 결과가 마지막 선언 기준으로 동작하는지 확인한다.
   - 예상 리소스: 짧은 실습 시간, JavaScript 실행 환경
   - 예상 효과: 동일 이름 함수 선언 시 뒤의 선언이 앞의 선언을 덮어쓴다는 점을 재현해 이해할 수 있다.
2. 객체 리터럴과 값/참조 비교 예제를 분리해 학습 노트를 만든다.
   - 구체적 스텝: 객체 리터럴 예제와 `Call by Value`, `Call by Reference` 예제를 각각 실행하고 결과 로그를 함께 기록한다.
   - 예상 리소스: 짧은 실습 시간, JavaScript 실행 환경, 메모 도구
   - 예상 효과: 기본형과 객체가 복사·참조되는 방식의 차이를 예제 결과로 명확히 구분할 수 있다.

## 원문 요약 (≤5문장)
이 글은 JavaScript의 기초 개념 세 가지를 간단한 예제로 정리한다. 먼저 동일한 이름의 함수를 중복 선언하면 아래쪽 함수가 위쪽 함수를 덮어쓴다고 설명한다. 다음으로 객체 생성은 Java식 생성자 호출보다 리터럴 형식을 사용한다고 소개한다. 마지막으로 기본형은 `Call by Value`, 객체는 `Call by Reference`로 동작해 값 변경 결과가 다르게 나타난다고 비교한다.

## 본문 포인트별 발췌
> JavaScript는 생성자라는 개념이 없다.
> 결론 : 이러한 경우 아래의 function이 위의 function을 덮어쓴다.
> 객체 생성의 경우 Java와 비슷하게 var member = new Member()와 같이 객체생성을 하지 않는다.
> 객체 생성의 경우 리터럴 형식을 사용한다.
> CallbyValue의 경우 각각의 하나의 값을 바라보지만, CallbyReference의 경우 객체의 주소를 같이 바라보고 있기 때문에 예제와 같이 한 곳의 값이 바뀌면 둘다 값의 변경이 이루어진다.
