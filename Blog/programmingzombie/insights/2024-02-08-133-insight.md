## 핵심 주장 (≤5)
1. GithubAction에서는 배포 실패로 표시되더라도 Elastic Beanstalk에서는 실제 배포가 정상 완료될 수 있다.
2. Spring Boot 애플리케이션은 초기 구동 시 데이터베이스 연결, JPA 엔티티 스캔, 의존성 주입 등으로 시작 시간이 길어져 헬스체크 실패가 잦아질 수 있다.
3. `einaregilsson/beanstalk-deploy` 액션의 기본 복구 대기 시간은 30초라서 초기화가 긴 서버에는 부족할 수 있다.
4. `wait_for_environment_recovery` 값을 늘리면 배포 후 헬스체크 실패 문제를 줄일 수 있다.
5. 장기적으로는 서버 최적화가 바람직하지만, 단기적으로는 복구 대기 시간을 늘리는 방식이 실용적인 대응책이다.

## 공개된 숫자·지표
- 기본 복구 대기 시간: 30초 (기간: 배포 직후 헬스 회복 대기, 출처: "Warning: Environment update finished, but health is Red and health status is Degraded. Giving it 30 seconds to recover...")
- `wait_for_environment_recovery` 기본값: 30sec (기간: GitHub Action 설정 기본값, 출처: "**wait_for_environment_recovery** 설정 값을 추가하면 된다고 적혀 있고, **기본 값은 30sec**으로 잡혀있는 것을 확인.")
- 조정한 복구 대기 시간: 180초 (기간: 설정 변경 후 재배포 시점, 출처: "넉넉히 180초로 증가 시켜서 다시 배포해보았다.")

## 언급된 도구·서비스
- GitHub Action: Elastic Beanstalk 배포 자동화 실행 환경으로 사용됨
- AWS Elastic Beanstalk: 서버 코드가 실제 배포되는 대상 환경으로 사용됨
- Spring Boot: 초기 구동 시 여러 초기화 작업이 발생하는 애플리케이션 프레임워크로 언급됨
- JPA: Spring Boot 초기화 과정에서 엔티티 스캔이 수행되는 요소로 언급됨
- `einaregilsson/beanstalk-deploy`: GitHub Action에서 Elastic Beanstalk 배포에 사용한 액션 플러그인
- AWS IAM 키: GitHub Secrets를 통해 배포 인증에 사용됨

## 언급된 다른 creator·앱
- rich-api-service: Elastic Beanstalk 배포 예시 코드에서 `application_name`과 `environment_name`으로 사용된 서비스명

## 복제 가능한 전술 (≤3)
1. Elastic Beanstalk 배포 액션의 환경 복구 대기 시간을 명시적으로 늘린다.
   - 구체적 스텝: GitHub Actions 워크플로에서 `einaregilsson/beanstalk-deploy@v21` 사용 구간을 찾고 `with` 블록에 `wait_for_environment_recovery` 값을 추가한 뒤 재배포한다.
   - 예상 리소스: GitHub Actions 워크플로 수정 1회, `einaregilsson/beanstalk-deploy` 액션
   - 예상 효과: 배포 직후 일시적인 `health Red`, `Degraded` 상태 때문에 GitHub Action이 실패로 끝나는 문제를 줄일 수 있다.
2. 배포 실패 표시와 실제 배포 성공 여부를 분리해서 점검한다.
   - 구체적 스텝: GitHub Action 로그에서 `Environment still has health Red` 오류를 확인한 뒤 Elastic Beanstalk 콘솔에서 새 애플리케이션 버전 반영 여부와 환경 상태를 직접 확인한다.
   - 예상 리소스: GitHub Action 로그, AWS Elastic Beanstalk 콘솔
   - 예상 효과: CI 실패 표시만 보고 배포 자체를 실패로 오판하는 일을 줄일 수 있다.
3. 초기 구동 시간이 긴 애플리케이션은 헬스체크 실패를 전제로 배포 설정을 조정한다.
   - 구체적 스텝: Spring Boot처럼 초기화 작업이 많은 서비스에서 배포 초기에 데이터베이스 연결, 엔티티 스캔, 의존성 주입으로 시간이 걸리는지 점검하고, 우선 복구 대기 시간을 늘린 뒤 이후 서버 최적화를 진행한다.
   - 예상 리소스: Spring Boot 애플리케이션, 배포 로그 확인 시간
   - 예상 효과: 단기적으로 배포 파이프라인의 불필요한 실패 표시를 줄이고, 장기 최적화 전까지 운영 마찰을 낮출 수 있다.

## 원문 요약 (≤5문장)
작성자는 GitHub Action과 Elastic Beanstalk를 연동한 자동 배포에서, 실제 배포는 성공했는데도 GitHub Action이 실패로 끝나는 문제를 겪었다. 원인은 배포 후 환경 헬스가 회복되기를 기다리는 시간이 기본 30초로 짧기 때문이라고 판단했다. 특히 Spring Boot는 초기 구동 시 데이터베이스 연결, JPA 엔티티 스캔, 의존성 주입 등으로 시작 시간이 길어질 수 있다고 설명한다. 그래서 `einaregilsson/beanstalk-deploy` 액션에 `wait_for_environment_recovery` 설정을 추가하고 대기 시간을 180초로 늘려 재배포했다. 그 결과 이후에는 실패가 발생하지 않았다고 정리한다.

## 본문 포인트별 발췌
> GithunAction + ElasticBeanstalk 조합으로 서버 코드 배포 자동화를 해놨는데, 이게 간헐적으로(매우 높은 확률로) 아래와 같이 흘러가다가 GithubAction 상에서 에러로 남아 있는게 아닌가..!
> GithubAction 상에는 에러로 남아있지만 실제로 ElasticBeanstalk을 들어가보면 아래와 같이 정상적으로 배포가 되었음을 볼 수 있었다.
> 더군다나 로그를 보면 헬스체크가 정상적으로 되길 기다리는 시간이 30초 밖에 주어지지 않는데 너무짠게 아닌가? 라는 생각이 들었다.
> **wait_for_environment_recovery** 설정 값을 추가하면 된다고 적혀 있고, **기본 값은 30sec**으로 잡혀있는 것을 확인.
> recovery 시간을 증가시켰더니 그 이후로는 실패가 되지 않는 것을 확인했다.
