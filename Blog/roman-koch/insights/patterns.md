# roman-koch 반복 패턴 분석

## 반복 사용 전술
- 아이디어 선검증: 코딩 전에 수요부터 확인하는 전술이 4번 등장한다. 랜딩 페이지, 사전등록, Twitter 질문, Reddit 관찰, ASO 키워드 조사로 “만들 가치가 있는 문제인가”를 먼저 거른다. 특히 2025-04, 2026-01 글에서는 인기·난이도·리뷰 수·매출 추정치를 함께 보고, 반응이 없으면 바로 폐기하는 식으로 개발 낭비를 줄인다.
- 빠른 MVP와 반복 출시: 완벽주의를 버리고 핵심 기능만 내놓는 전술이 4번 반복된다. 2025-02에서는 80/20 규칙으로 기능을 줄이고, 2026-01 회고에서는 2025년에 8개 제품을 출시하며 시장 데이터를 학습 자산으로 쌓았다. 빠른 출시는 목표가 아니라 학습 루프의 시작점으로 다뤄진다.
- 배포 우선 사고: 마케팅·ASO·가시성을 제품 일부로 보는 전술이 5번 이상 나온다. 단순히 앱을 만드는 것이 아니라 아이콘, 스크린샷, 스토어 노출, 소셜 채널, 자체 오디언스까지 묶어 설계한다. 2026-02 글에서는 이 병목이 “engineering”에서 “visibility”로 이동했다고 명시한다.
- 실제 사용 맥락 중심 설계: 기능 추가보다 실제 스트레스 상황에서의 사용성을 다듬는 전술이 3번 보인다. Blood Pressure 앱 사례에서는 입력 탭 수, 히스토리 가독성, 공유 흐름처럼 평소엔 사소해 보이는 요소를 핵심 가치로 재정의한다.

## 일관되게 사용하는 도구
- ASO tools: 가장 일관된 리서치 도구다. 키워드 인기, 난이도, 경쟁 강도, 스토어 노출 가능성을 판단하는 데 3개 글에서 반복 사용된다.
- Reddit: 문제의 진짜 고통과 반복 질문을 찾는 검증 채널이다. 초기 아이디어 반응 확인과 사용자 언어 수집에 모두 쓰인다.
- Twitter/X: 낯선 사람의 정직한 반응을 받는 검증 채널이자, 빌드 과정을 퍼뜨리는 배포 채널로 이중 사용된다.
- Cursor: 2026년부터는 빌드 속도를 높이는 핵심 제작 도구로 부상한다. 단, 저자는 도구 자체보다 문서화된 컨텍스트가 성과를 만든다고 본다.
- Telegram: 고객 지원을 즉시 처리하기 위한 운영 도구다. 혼자 여러 앱을 운영하는 상황에서 대응 속도를 차별점으로 만든다.

## 반복되는 수익화 패턴
- 고통이 큰 문제 중심 수익화: 그냥 “실제 문제”가 아니라 많은 사람이 적극적으로 해결하려는 문제를 고른다. SmokeFree App은 약 1,480 다운로드와 약 $700 매출로, 다른 실험 앱들보다 훨씬 강한 성과를 냈다.
- 스토어 데이터 기반 수익 판단: 월 매출 추정이 $20,000 이상이면 강한 신호, $5,000 미만이면 위험 신호로 본다.
- 범용 구독 회피: 2026-02부터는 포화 시장에서 고정 구독보다 결과 기반 과금, 사용량 기반 과금, 번들, B2B가 더 맞는 구조라고 방향을 튼다.

## 성장 궤적
2025년 초에는 초보 인디 개발자로서 검증 부족, 완벽주의, 마케팅 소홀 같은 실패를 복기하는 단계였다. 2025년 중반에는 검증 프로세스와 지원 시스템처럼 운영 레이어를 체계화하기 시작했다. 2026년 초에는 연간 8개 제품 실험의 결과를 바탕으로 ASO, 배포, 수익화 구조를 더 정교하게 해석하고, Cursor 기반 제작 워크플로우까지 정립한다. 이후 2026-03에는 자신의 제품을 실제 위기 상황에서 써보며 사용자 관점의 밀도를 높였고, 2026-04에는 다시 “무엇이 팔리나”뿐 아니라 “무엇을 만들고 싶은가”를 회복하는 쪽으로 사고가 확장된다. 즉, 시행착오 중심의 초반기에서 데이터 기반 운영기로 갔다가, 최근에는 데이터와 동기 사이의 균형을 다시 잡는 단계로 진화했다.
