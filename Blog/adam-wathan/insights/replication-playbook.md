# adam-wathan 전략 복제 플레이북

## 복제할 핵심 전략 (Top 3)
- 작은 유료 제품으로 시작해 큰 사업으로 확장: Adam은 처음부터 대형 SaaS를 만들지 않고 작은 유료 책과 코스로 출시 경험을 쌓았고, 이후 Refactoring UI, Tailwind CSS, Tailwind UI로 확장했다. TemperStone도 처음부터 "큰 앱"을 만들기보다 1개 문제를 푸는 작고 유료인 산출물로 시작해야 한다. 기대 결과는 출시 학습 속도 증가, 심리적 진입장벽 감소, 초기 현금흐름 확보다.
- 공개적으로 만들고 수요 신호를 제품 후보로 전환: Adam은 이메일 리스트, 트위터, 라이브스트리밍, 무료 글/스크린캐스트로 작업 과정을 계속 노출했고, KiteTail 작업 공개가 Tailwind Labs의 출발점이 됐다. TemperStone도 제품 완성 후 홍보가 아니라 제작 중 질문, 반응, 재사용 패턴을 수요 신호로 해석해야 한다. 기대 결과는 제품 아이디어 정확도 상승, 출시 전 수요 축적, 커뮤니티 형성이다.
- 오픈 자산으로 배포를 만들고 프리미엄 패키지로 수익화: Adam은 Tailwind CSS를 오픈소스로 키운 뒤 Tailwind UI 같은 상업 제품으로 연결했고, 책도 단일 PDF가 아니라 스크린캐스트·예제 프로젝트를 묶어 평균 판매가를 끌어올렸다. TemperStone도 무료 배포 채널과 유료 업셀을 분리 설계해야 한다. 기대 결과는 낮은 진입장벽과 높은 ARPU를 동시에 확보하는 것이다.

## 어떤 앱을 만들 것인가 (앱 종류 선정 기준)
- Adam의 선정 패턴은 "내가 여러 프로젝트에서 반복해서 쓰는 것", "사람들이 작업 과정을 보며 계속 질문하는 것", "서로 다른 프로젝트에도 이식 가능한 것"에 집중된다.
- 성공 사례는 Refactoring to Collections 같은 교육형 디지털 제품, Advanced Vue Component Design 같은 짧은 집중 제작 코스, Tailwind CSS처럼 반복 재사용된 내부 스타일 시스템, Tailwind UI처럼 오픈 자산 위의 프리미엄 상업 제품이다.
- 실패 또는 비효율 사례는 Digest처럼 본 프로젝트는 접혔지만, 그 부산물인 스타일 시스템만 살아남은 경우다. 이 패턴은 "실패한 앱도 부산물을 남기면 된다"가 아니라 "살아남는 공통층만 제품 후보"라는 뜻이다.
- Adam은 컴포넌트보다 유틸리티가 더 portable하다고 봤다. TemperStone도 특정 서비스에 강하게 묶인 완성형 앱보다, 여러 프로젝트에서 그대로 재사용되는 도구층, 템플릿층, 워크플로층을 먼저 찾아야 한다.
- 아이디어 발굴 방법:
- 최근 6개월 내 TemperStone이 만든 프로젝트에서 반복 복붙한 코드, 운영 문구, 데이터 구조, 배포 루틴을 전부 적는다.
- 그중 최소 두 개 이상의 프로젝트에서 살아남은 것만 남긴다. Adam은 Digest 스타일을 최소 4~5개 프로젝트에 가져갔다.
- 작업 공개 채널에서 질문이 반복되는 주제를 분리한다. Adam은 라이브스트리밍 중 CSS 질문이 Tailwind 공개의 직접 계기였다.
- 첫 제품은 "독립 앱"보다 "작은 유료 도구", "교육형 산출물", "설정/템플릿/키트", "오픈소스+유료 업셀" 중 하나로 고른다.
- 선정 기준:
- 특정 고객군이 바로 이해할 수 있는 문제인가
- 2개월 미만 집중 제작으로 끝낼 수 있는가
- 무료 버전과 유료 버전을 자연스럽게 나눌 수 있는가
- 서로 다른 프로젝트에서도 통하는 project-agnostic 구조인가

## 얼마나 빠르게 출시할 것인가 (대량 생산 파이프라인)
- Adam의 핵심은 오래 끌지 않는 것이다. Advanced Vue Component Design은 아이디어부터 출시까지 2개월 미만이었다.
- 반대로 Test-Driven Laravel은 2016년 11월 얼리 액세스로 시작했지만 최종 레슨 공개가 2018년 1월이었고, Adam은 이 경험을 통해 프리셀 방식이 자신에게 맞지 않는다고 판단했다.
- 책 제작에서는 본문 9주, 스크린캐스트 1주로 분리했고, 이미 갖고 있던 샘플 프로젝트를 최고가 패키지에 묶어 수익을 키웠다.
- Tailwind는 2017년 6~7월 이후 약 2~3개월 동안 오픈소스 공개 가능 수준까지 다듬었다.
- Tailwind UI 초기 액세스는 자가 마감 직전 36시간 연속 작업까지 감수할 정도로 출시 속도를 우선했다.
- TemperStone용 출시 파이프라인:
- 1단계: 1주 내 문제 한 줄 정의, 랜딩 페이지 공개, 이메일 수집 시작
- 2단계: 2주 내 무료 샘플 1개 또는 공개 데모 1개 배포
- 3단계: 4~8주 내 첫 유료 버전 출시
- 4단계: 출시 전까지 진행 업데이트, 샘플, 무료 글/영상, 출시일 공지 반복
- 5단계: 유료 버전은 단일 상품이 아니라 기본형과 상위 패키지로 설계
- 대량 생산 방법론:
- 새 제품을 매번 처음부터 만들지 말고, 기존 프로젝트 부산물에서 portable layer를 뽑아낸다.
- 추상화는 미리 하지 말고 반복이 확인될 때만 한다. Adam은 CSS에서도 utility로 먼저 만들고 반복될 때만 컴포넌트를 추출했다.
- 기술 글, 무료 스크린캐스트, 라이브스트림, 제품 문서, 유료 제품을 하나의 생산 체인으로 묶는다.
- Adam이 실제로 보여준 속도 지표:
- 2018년 Tailwind CSS 릴리스 17회
- 2018년 라이브스트림 18회
- 2018년 Full Stack Radio 신규 에피소드 25개
- 2018년 장문 아티클 2개

## 출시 후 무엇을 측정하고 유지할 것인가 (AdMob 수익 모니터링)
- 원문 기준 Adam Wathan은 AdMob·Google Ads 운영 사례를 말하지 않는다. 따라서 TemperStone이 광고 기반 앱을 만들더라도, 이 섹션에서 Adam식으로 직접 복제할 수 있는 것은 광고 최적화가 아니라 "수요·배포·업셀" 지표 운영 방식이다.
- Adam이 실제로 본 핵심 지표:
- 이메일 리스트 규모: 런치 전 1,500명
- 소셜 배포 자산: 트위터 팔로워 약 4,000명
- 3일 런치 매출: $61,392
- 평균 판매가: $67.84
- 11개월 누적 매출: $138,835
- 출시 후 지속 매출: 월 $1,000-$2,000
- Tailwind CSS 설치 수: 2020년 기준 1,000만 회 돌파
- Tailwind UI 매출: 출시 약 5개월 만에 200만 달러 돌파 직전
- Tailwind Labs 매출: 2년 미만에 400만 달러 초과
- TemperStone이 유지해야 할 모니터링 항목:
- 사전 수요: 이메일 구독자 수, 무료 콘텐츠 반응, 공개 작업에서 반복되는 질문 수
- 출시 효율: 첫 3일 매출, 패키지별 판매 비중, 평균 판매가
- 장기 유지: 출시 후 월간 반복 매출, 제품별 설치/활성/커뮤니티 증가
- 배포 자산: 문서 방문, 무료 콘텐츠 소비, 커뮤니티 회원 수, SNS 팔로워 증가
- 판단 기준:
- 첫 제품은 Adam처럼 "작아도 바로 판다"가 중요하다. 매출 절대액보다 출시 성공 경험과 고객 반응 확보가 우선이다.
- 상위 패키지가 평균 판매가를 끌어올리는지 본다. Adam은 단일 $29 상품만 팔았으면 같은 고객 수에서 $26,245였지만, 실제 평균 판매가는 $67.84였다.
- 출시 후에도 월간 유지 매출이 생기는지 본다. Adam의 첫 책은 11개월 후에도 월 $1,000-$2,000를 만들었다.
- 광고 전략에 대한 TemperStone용 해석:
- 광고 기반 앱을 하더라도 Adam식 핵심은 광고 최적화가 아니라 자체 배포 채널 확보와 프리미엄 업셀 설계다.
- 즉, AdMob 수익만 보는 구조는 Adam식과 거리가 멀다. 무료 트래픽 상품이 있다면 광고는 보조 수익으로 두고, 주 수익원은 유료 패키지 또는 프리미엄 제품으로 설계하는 편이 더 가깝다.

## 주의사항 & 함정
- 프리셀을 길게 끌지 말 것: Adam은 범위를 유연하게 바꾸기 어렵고 자신처럼 낙관적 범위 설정을 하는 사람에게 프리셀이 맞지 않았다고 회고했다.
- 제품을 너무 크게 정의하지 말 것: Advanced Vue Component Design은 2개월 미만으로 끝냈지만, Test-Driven Laravel은 얼리 액세스부터 최종 완료까지 지나치게 길어졌다.
- 특정 프로젝트에만 맞는 완성형 앱을 바로 제품화하지 말 것: Adam은 컴포넌트보다 유틸리티가 더 portable하다고 확인했고, Tailwind도 서로 다른 프로젝트에 동시에 맞춰보며 범용성을 얻었다.
- 무료 배포 자산 없이 유료화부터 하지 말 것: Adam은 이메일, 트위터, 무료 글, 무료 스크린캐스트, 라이브스트리밍, 오픈소스를 모두 수요 축적 채널로 썼다.
- 광고 기반 모델을 주력으로 착각하지 말 것: 원문에서 Adam의 큰 성과는 광고 최적화가 아니라 직접 판매, 패키징, 오픈소스 기반 상업화에서 나왔다.
- TemperStone 컨텍스트에서 특히 주의할 점:
- "앱을 많이 만든다"보다 "재사용 가능한 공통층을 빨리 식별한다"가 우선이다.
- 팀이 작다면 커뮤니티 운영 채널을 처음부터 너무 많이 벌리지 말고, 질문이 쌓일 때 공식 아카이브 채널을 연다.
- 기술적 차별화는 기능 수가 아니라 composability와 portability에서 나와야 한다.
