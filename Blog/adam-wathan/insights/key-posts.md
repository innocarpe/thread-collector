## 1. Tailwind CSS: From Side-Project Byproduct to Multi-Million Dollar Business
- **파일**: 2020-08-02-tailwindcss-from-side-project-byproduct-to-multi-mullion-dol-insight.md
- **URL**: https://adamwathan.me/tailwindcss-from-side-project-byproduct-to-multi-mullion-dollar-business
- **핵심 주장**: 실패한 사이드 프로젝트의 부산물이라도 여러 실전 프로젝트에서 반복 재사용되면 독립 제품의 씨앗이 될 수 있다. 작업 과정을 공개하면 예상 못 한 수요 신호가 생기고, 오픈소스는 그 수요를 증폭하는 배포 채널이 된다. Tailwind는 재사용 가능한 유틸리티 층만 남기고 범용성을 높인 뒤, 상업 제품 Tailwind UI로 이어지며 큰 사업이 됐다.
- **주목 이유**: 설치 1,000만 회, Tailwind UI 출시 약 5개월 만에 200만 달러 근접, Tailwind Labs 2년 미만 400만 달러 초과라는 수치가 모두 공개된 드문 창업 기록이다. "부산물 → 공개 작업 → 오픈소스 → 유료 제품" 경로가 가장 선명하게 정리된 글이다.

## 2. The $61,392 Book Launch That Let Me Quit My Job
- **파일**: 2017-04-21-the-book-launch-that-let-me-quit-my-job-insight.md
- **URL**: https://adamwathan.me/the-book-launch-that-let-me-quit-my-job
- **핵심 주장**: 작은 디지털 제품으로 먼저 출시 경험을 쌓고, 이메일 리스트를 중심으로 샘플과 진행 상황을 반복 공유하면 작은 청중으로도 강한 런치를 만들 수 있다. 단일 상품보다 티어드 패키징이 평균 판매가를 크게 올린다. 런치 전 공유 요청, 미구매자 후속 이메일, 후기 활용 같은 고전 전술도 여전히 유효하다.
- **주목 이유**: 3일 런치 매출 61,392달러, 평균 판매가 67.84달러, 출시 11개월 누적 138,835달러까지 전부 공개돼 있다. 솔로 크리에이터가 이메일 1,500명과 트위터 4,000명 규모로 퇴사 가능한 현금흐름을 만든 방식이 구체적이다.

## 3. Going Full-Time on Tailwind CSS
- **파일**: 2018-12-28-going-full-time-on-tailwind-css-insight.md
- **URL**: https://adamwathan.me/going-full-time-on-tailwind-css
- **핵심 주장**: 프로젝트가 빠르게 채택되기 시작하면 창업자는 기능 개발만이 아니라 문서, 교육, 커뮤니티 인프라, 수익화 구조를 함께 설계해야 한다. Tailwind의 다음 단계는 v1.0 출시보다도 공식 문서 보강, 무료 코스, 커뮤니티 허브 정비로 진입장벽을 낮추는 일이었다. 오픈소스의 지속 가능성은 기존 교육 비즈니스, 후원, 프리미엄 제품을 조합해 만든다.
- **주목 이유**: 릴리스 30회, 기여자 77명, npm 설치 약 70만 회, 스타 8,000+, 슬랙 1,100+, 트위터 10,000+ 등 전업 전환 판단의 근거가 수치로 제시된다. 오픈소스 프로젝트를 "일"로 전환하는 운영 청사진이 담겨 있다.

## 4. 2018 Year in Review
- **파일**: 2018-12-27-2018-year-in-review-insight.md
- **URL**: https://adamwathan.me/2018-year-in-review
- **핵심 주장**: 장기 프로젝트의 비효율을 겪은 뒤에는 아이디어에서 출시까지의 압축된 리듬이 훨씬 중요해진다. 프리셀은 범위 수정이 어려운 창작자에게 오히려 독이 될 수 있고, "충분함"을 기준으로 사업을 설계해야 지속된다. 동시에 글, 팟캐스트, 발표, 라이브스트림 같은 공개 활동은 장기적으로 강한 복리 자산이 된다.
- **주목 이유**: 6,000부 이상 판매한 Refactoring UI 론치, Tailwind 17회 릴리스와 67만 설치, 팟캐스트 200만 다운로드, 18회 라이브스트림 등 1년 운영 데이터가 촘촘하다. 개인 사업 포트폴리오를 어떻게 조율했는지 보여주는 밀도 높은 회고다.

## 5. CSS Utility Classes and "Separation of Concerns"
- **파일**: 2017-08-07-css-utility-classes-and-separation-of-concerns-insight.md
- **URL**: https://adamwathan.me/css-utility-classes-and-separation-of-concerns
- **핵심 주장**: CSS 설계의 본질은 "관심사의 분리"가 아니라 HTML과 CSS 중 무엇이 무엇에 의존하는가의 문제다. 콘텐츠 전용 클래스보다 재사용 가능한 패턴과 유틸리티를 조합하는 편이 중복을 줄이고 일관성을 높인다. 컴포넌트는 처음부터 추상화하지 말고, 유틸리티 조합이 반복될 때만 추출하는 것이 낫다.
- **주목 이유**: GitLab, Stripe, GitHub 등 실제 서비스의 CSS Stats 수치를 근거로 "새 CSS를 계속 쓰는 비용"을 설득한다. Tailwind 철학의 원형이 가장 명확하게 서술된 글이라 제품 철학과 실행 전술을 동시에 읽을 수 있다.

## 6. Composing the Uncomposable with CSS Variables
- **파일**: 2020-09-18-composing-the-uncomposable-with-css-variables-insight.md
- **URL**: https://adamwathan.me/composing-the-uncomposable-with-css-variables
- **핵심 주장**: 유틸리티 시스템이 순수 CSS의 표현력을 따라가려면, 원래 조합이 불가능한 속성도 클래스 단위로 분해할 수 있어야 한다. CSS 변수는 완성된 값뿐 아니라 부분 값도 담을 수 있기 때문에 `transform`, 색상 opacity, `font-variant-numeric` 같은 속성에 조합 가능한 슬롯을 만들 수 있다. 핵심은 기본 클래스에서 no-op 값을 선언하고, 후속 유틸리티는 속성 자체가 아니라 변수만 덮어쓰게 만드는 것이다.
- **주목 이유**: 단순한 팁 글이 아니라 Tailwind의 핵심 구현 철학을 설명하는 설계 문서에 가깝다. "복잡한 CSS를 어떻게 제품화 가능한 유틸리티 시스템으로 환원하는가"에 대한 가장 복제 가능한 기술 글이다.

## 7. Persistent Layout Patterns in Next.js
- **파일**: 2019-10-17-persistent-layout-patterns-in-nextjs-insight.md
- **URL**: https://adamwathan.me/2019/10/17/persistent-layout-patterns-in-nextjs
- **핵심 주장**: Next.js는 기본적으로 페이지 전환 때 전체 UI를 다시 렌더링하기 때문에, SPA다운 지속 UI 경험을 얻으려면 별도 패턴이 필요하다. `_app.js` 공통 레이아웃, URL 기반 분기, 정적 `layout`, `getLayout` 함수 패턴을 비교해 보면 페이지별 `getLayout`이 가장 선언적이고 유연하다. 중요한 포인트는 레이아웃 인스턴스가 DOM 상 같은 위치에 유지되도록 트리를 설계하는 것이다.
- **주목 이유**: 단순 문제 제기가 아니라 4가지 구현 패턴의 장단점을 명확히 비교해 곧바로 적용할 수 있다. 많은 Next.js 팀이 실제로 채택한 설계 패턴의 출발점으로 자주 인용되는 글이다.

## 8. Renderless Components in Vue.js
- **파일**: 2018-03-22-renderless-components-in-vuejs-insight.md
- **URL**: https://adamwathan.me/renderless-components-in-vuejs
- **핵심 주장**: 복잡한 UI 동작은 렌더링과 분리해 상태와 행동만 관리하는 렌더리스 컴포넌트로 만들 수 있다. 부모는 scoped slot을 통해 HTML을 완전히 통제하고, 자식은 데이터·액션·바인딩을 제공하는 구조가 가장 유연하다. 동일한 동작을 여러 레이아웃에 재사용해야 할 때 이 패턴의 가치가 커진다.
- **주목 이유**: 추상적 개념이 아니라 태그 입력 컴포넌트를 단계적으로 분해하며 API 설계법까지 보여준다. 라이브러리형 UI 설계와 커스터마이징 전략을 함께 배울 수 있는 실전 글이다.

## 9. Don't Use Em for Media Queries
- **파일**: 2017-11-30-dont-use-em-for-media-queries-insight.md
- **URL**: https://adamwathan.me/dont-use-em-for-media-queries
- **핵심 주장**: Safari 확대 환경에서는 `px`보다 `em` 미디어 쿼리가 더 늦게 발동하며, 이 때문에 레이아웃 붕괴 시점이 실제 읽기 경험과 어긋난다. 따라서 당시 기준으로는 미디어 쿼리에 `px`를 쓰는 쪽이 브라우저 간 일관성이 높다. 다만 접근성을 해치는 루트 폰트 고정은 더 나쁜 절충안일 수 있다.
- **주목 이유**: 400px, 25em, 150% 줌, Safari 900px 발동 같은 구체 값으로 흔한 CSS 통념을 반박한다. 실무에서 반복 인용되는 "작은 브라우저 검증 글"의 좋은 예다.

## 10. The Trick to Understanding Scoped Slots in Vue.js
- **파일**: 2018-04-19-the-trick-to-understanding-scoped-slots-in-vuejs-insight.md
- **URL**: https://adamwathan.me/the-trick-to-understanding-scoped-slots-in-vuejs
- **핵심 주장**: scoped slots는 함수 prop처럼 이해하면 진입 장벽이 크게 낮아진다. 낯선 프레임워크 개념을 익숙한 프로그래밍 추상화로 번역하는 순간 학습 속도가 빨라진다. 이 관점은 이후 더 복잡한 컴포넌트 설계 패턴을 받아들이는 발판이 된다.
- **주목 이유**: 분량은 짧지만 "어려운 개념을 어떻게 가르칠 것인가"에 대한 교육 전략이 선명하다. Adam Wathan식 기술 교육 콘텐츠의 축약판처럼 읽히는 글이다.
