# weyoume 전략 복제 플레이북

## 복제할 핵심 전략 (Top 3)
- 개인 루틴을 공개 실험으로 바꾸기: weyoume는 15주차부터 26주차까지 거의 매주 같은 포맷으로 체중·운동·식단 변화를 기록했다. TemperStone도 창업자 본인의 반복 문제를 주간 단위 공개 로그로 운영하면, 콘텐츠와 제품 아이디어를 동시에 확보할 수 있다.
- 아주 좁은 문제를 즉시 해결해 짧게 발행하기: `Non-constant range...`, `Show Live Issues`, `Build input file cannot be found`처럼 한 가지 오류와 한 가지 해결책만 다루는 글을 발행했다. 복제 가능성이 높은 이유는 큰 제품보다 작은 문제 해결이 더 빨리 출시되고 검색 유입을 만들기 쉽기 때문이다.
- 정체 구간에서 변수 하나만 바꿔 재실험하기: 22주차~25주차 정체 뒤 26주차에 저녁 식단과 계단 오르기 방식만 바꿔 다시 감량을 만들었다. TemperStone도 앱이 정체되면 한 번에 하나의 변수만 바꾸는 운영으로 학습 속도를 높일 수 있다.

## 어떤 앱을 만들 것인가 (앱 종류 선정 기준)
- creator의 앱 선정 패턴: 이 묶음에서 직접 보이는 패턴은 `자기 문제`, `반복 사용`, `설명이 짧은 문제`, `검색어가 선명한 문제`다. 다이어트 글은 본인이 매일 반복하는 루틴을 다뤘고, 개발 글은 본인이 실제로 부딪힌 Xcode/Swift 문제를 다뤘다.
- 성공 패턴: 두부, 계란, 단백질빵, 닭안심처럼 “기존 행동을 완전히 바꾸지 않고 대체 가능한 것”이 반복해서 채택됐다. 앱도 같은 기준으로 가야 한다. 사용자의 기존 행동을 갈아엎는 앱보다, 이미 하던 일을 더 쉽게 하게 만드는 초경량 유틸리티가 맞다.
- 실패·경고 패턴: 22주차와 23주차에는 같은 루틴으로 추가 감량이 멈췄다. TemperStone도 기능이 늘어도 사용 패턴이 안 바뀌면 성장이 멈출 수 있으므로, “좋아 보이는 기능”보다 “반복 사용을 더 자주 만드는 기능”을 우선해야 한다.
- 아이디어 발굴 방법 1: 창업팀이 지난 30일간 반복해서 겪은 불편을 적는다. 조건은 “한 문장으로 설명 가능”, “하루 또는 매주 반복”, “직접 쓸 의향 있음”이다.
- 아이디어 발굴 방법 2: 해결 글이 5문장 안에 끝나는 문제를 우선한다. weyoume의 개발 글들은 원인과 해결책이 매우 짧다. 앱도 설정 하나, 경로 확인 하나, 기록 하나로 해결되는 문제를 우선 선정한다.
- 아이디어 발굴 방법 3: 검색형 주제와 루틴형 주제를 동시에 운영한다. 검색형은 Xcode 오류 같은 문제 해결 앱, 루틴형은 기록·체크·측정 앱이다.
- TemperStone에 맞는 1차 후보군:
- iOS 개발자용 초협소 유틸리티 앱: 빌드 오류 체크리스트, Xcode 설정 점검, 경로/파일 메타데이터 확인 같은 문제 해결형.
- 자기관리 기록 앱: 주간 체중·식단·운동처럼 동일 포맷을 반복 기록하고 변화만 남기는 루틴형.
- 대체 선택 앱: 두부/계란/닭안심처럼 “대체재”를 찾는 구조를 본떠, 기존 선택을 더 나은 선택으로 바꿔주는 비교형 앱.

## 얼마나 빠르게 출시할 것인가 (대량 생산 파이프라인)
- creator의 다작 방식에서 직접 보이는 수치:
- 다이어트 주간 리포트는 15주차부터 26주차까지 이어졌다.
- 식단 실험 글은 2025-02-20부터 2025-03-21 사이에 7편이 나왔다.
- 개발 트러블슈팅 글은 2025-09-27, 2025-09-28, 2025-10-01처럼 문제 발생 근처 시점에 연속 발행됐다.
- 해석: weyoume식 생산성의 핵심은 “대작을 적게”가 아니라 “짧은 단위 결과물을 자주”다.
- TemperStone 실행 파이프라인:
- 1단계 아이디어 수집: 팀 내부에서 반복 문제를 모은다.
- 2단계 즉시 검증: 팀원이 직접 3일~7일 써본다.
- 3단계 초소형 출시: 기능 하나만 넣어 MVP를 만든다.
- 4단계 공개 기록: 출시 이유, 해결 문제, 바꾼 변수, 다음 실험을 바로 글로 남긴다.
- 5단계 반복: 반응이 없으면 전체 재설계보다 변수 하나만 바꾼다.
- 대량 생산 원칙:
- 하나의 앱은 하나의 문제만 해결하게 만든다.
- 하나의 릴리스는 하나의 가설만 검증하게 만든다.
- 앱 출시와 블로그 발행을 한 묶음으로 운영한다.
- 자동화 도구: 원문에 직접 나온 도구는 Xcode, Swift 6, 시뮬레이터, 블로그다. TemperStone은 최소한 `앱 빌드`, `출시 노트 생성`, `문제 해결 글 발행`을 같은 흐름으로 묶어야 한다.

## 출시 후 무엇을 측정하고 유지할 것인가 (AdMob 수익 모니터링)
- 이 인사이트 묶음에는 AdMob·Google Ads 운영 수치와 세팅 방식이 직접 나오지 않는다.
- 다만 creator의 운영 습관에서 직접 보이는 것은 `주간 기록`, `전후 비교`, `정체 시 변수 변경`, `일별/주별 추세 확인`이다. TemperStone의 AdMob 운영도 이 방식으로 복제하는 것이 맞다.
- 운영 방법:
- 앱별로 리포트를 분리해 본다.
- 일별 수치보다 주간 추세를 우선 본다.
- 광고 성과를 볼 때는 같은 주간의 사용량 변화와 같이 본다.
- 수익 판단 기준:
- 출시 직후 일시 상승보다 2주 이상 같은 방향의 추세가 있는지 본다.
- 노출이 늘었는데 유지율이 떨어지면 광고 배치가 과한 것으로 본다.
- 정체가 오면 광고 위치, 빈도, 앱 온보딩 중 한 변수만 바꿔 본다.
- 광고 전략:
- 반복 사용 앱에 광고를 붙이고, 1회성 해결 앱은 검색 유입과 다른 앱 유입 채널로 본다.
- 문제 해결형 앱은 광고 수익만 보지 말고 후속 앱으로 보내는 관문 역할도 측정한다.
- 원문 직접 근거가 없는 영역이므로, 숫자 임계값은 TemperStone이 실제 데이터로 별도 정의해야 한다.

## 주의사항 & 함정
- creator가 경고한 것들:
- 같은 루틴도 어느 시점에는 더 이상 추가 성과를 만들지 않는다. 22주차와 23주차는 정체였다.
- 주말 음주처럼 예외 행동은 주간 성과를 왜곡한다. TemperStone도 마케팅 이벤트나 외부 유입 급증일을 분리해서 봐야 한다.
- 정체를 깨려면 변수 하나만 바꿔야 한다. weyoume는 26주차에 저녁 식단과 계단 방식 변화를 명시했다.
- TemperStone 컨텍스트 주의점:
- 창업자 본인이 안 쓰는 앱은 이 전략과 맞지 않는다.
- 기능을 넓히면 weyoume식 검색형/반복형 강점이 사라진다.
- AdMob 운영은 이 묶음에서 직접 확인되지 않았으므로, 광고는 출시 후 별도 실험 트랙으로 관리해야 한다.
- 블로그는 홍보물이 아니라 학습 로그여야 한다. weyoume 글의 강점은 솔직한 기록과 전후 비교다.
