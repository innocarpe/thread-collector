# 바이브 코딩 팁(팁칸이 없어서 여기다가 적습니다)

철저하게 개인적인 사용후기 입니다.
1. 백엔드는 CODEX가 최고 입니다. 시간 좀 오래걸리지만 버그 없이 코드 깔끔하게 쓰려면 코덱스 쓰세요...
2. 프론트의 경우 IDE 는 안티그래비티 사용하시고, 모델은 제미니 pro가 가장 디자인 깔끔하게 나왔었습니다. 
3. 실전 팁은 "무조건" git hub 를 사용하세요!
​
그럼 독자분이 비개발자라는것을 가정하고 이야기 하겠습니다. 
왜 깃허브를 써야하나요?
1. AI는 기억력이 좋지 않습니다. 바이브 코딩을 하시다보면 갑자기 [압축]이라는 것을 진행합니다. 그럴때마다 지금까지 무슨 작업을 했는지, 이제 무엇을 해야하는지 이전보다 잘 알지 못하게 됩니다. 
2. 바이브 코딩을 하다보면 비개발자가 아닌 경우에 ai가 코딩을 정말 내가 원하는 대로 하고 있는지, 내가 원하는 기능을 잘 구현하고 있는지 모른채로 그냥 엔터 연타만 갈기고 있습니다. 그러다가 나중에 최종 결과물을 확인하게 되면 내가 요구하지도 않았던 기능을 개발하거나, 전혀 다른 기능을 구현하는경우가 굉장히 많습니다. 이때, 어느 시점으로 돌아가서 다시 개발을 시작해야하는지 알기 위해서는 [git hub]를 통해 작업 내역을 역추적하고, 그 시점으로 롤백해서 다시 작업을 해야할 일이 생기실겁니다. 아마 그렇지 않은 경우 거의 대부분 '아 그냥 처음 부터 다시 개발하자' 라는 식으로 시도를 많이 하게 되실겁니다. 
​
그럼 어떻게 써야하나요?
일단 자세한건 지피티에게 물어보세요. 
크게 알아야할건 커밋, 푸시, 브랜치, Pull Request입니다.
​
1. 커밋, 푸시
-> 커밋은 쉽게 말해 지금까지 한 작업들을 남기는 작업 기록입니다. 
-> 푸시는 지금까지한 작업 내용(코드) + 작업기록을 웹(깃허브)에 올리는 겁니다. 
​
2. 브랜치(이게 가장 사실 중요함)
-> 브랜치란 직역하면 그대로 나뭇가지 같은겁니다. 나무의 중심줄기(main)이 있다면, 거기에 새로운 기능을 바로 붙일 시 버그가 나거나 오류가 발생할 수 있습니다. 그래서 main 이라는 곳에 계속 하나씩 하나씩 새로운 기능을 추가할때마다 나무가지를 하나 따와서, 거기다가 새로운 기능을 붙인 이후에 main(나무의 중심 줄기)에다가 붙이는겁니다. 이렇게 해야 어디서 버그가 났는지, 또는 특정 기능으로 롤백할때 용이합니다. 특히 이미 서비스가 배포된 상황이라면 더더욱 갑자기 운영되던 서비스가 다운되는것을 막기 위해 이 브랜치 전략이 꼭 필요하실겁니다. 마치 실 서비스의 복제본을 하나 새로 만들어놓고, 새로운 기능을 미리 붙여보고 문제없으면 다시 배포하는 식이라고 생각하시면됩니다. 
​
이것도 비개발자에겐 그럼 어려우실겁니다. 그래서 codex나 claudecode에게 "Github cli를 설치해줘" 라고 일단 말하세요. 
그리고 이렇게 말하세요. "브랜치를 3개로 만들어줘. 1. main , 2. feature 3. hotfix"  이렇게 3개의 브랜치만 써도 충분할겁니다. 말 그대로 main은 "항상 배포 가능한 상태" 를 의미합니다. 최종 결과물이죠. feature는 말그대로 기능입니다. 기능 추가할때 쓰시면 됩니다. hotfix는 버그 고칠때 쓰는 겁니다. 그래서 이렇게 3개 브랜치로 나누어서 기능 추가하고, 버그 고치시면서 main에다가 계속 추가하시면 됩니다. 
​
3. Pull Request는요?
PR은 main 브랜치에 병합하기전에 한번 확인하는겁니다. 쉽게 말해 이런 순서입니다. feature/hotfix 브랜치 만들기-> ai가 코딩 → PR → main 이걸 그리고 그냥 ai에게 입력해두면서 이해하려고 하시지 마시고 그냥 직접해보시기 바랍니다. 하시면서 이해하는게 더 빠르실거에요. 
​
솔직히 개발새발 쓴거 같은데 개발자분이 읽으시면 굉장히 쪽팔릴만한 글이지만 조금이라도 도움이 되시면 좋겠습니다. 핵심은 바이브 코딩 "공부"를 하려고 하시지 마시고, 그냥 가서 써보세요. 깃허브 계정 연동부터  ai가 시키는대로 그냥 하면됩니다.  실행력이 답니다 여러분!
